# R3.3 e2e 入 CI 门禁（nightly，全 mock 模式确定性子集）

> 状态：落地中——首次 GitHub 实跑发现链路 2 遗漏（见 §6），已修复，待二次实跑确认全绿。
> 关联审计：`docs/reviews/2026-07-02-repo-audit-and-roadmap.md` §4 T3.3（G2：e2e 零门禁）。

## 1. 背景（为什么做）

R1/R2/R3.1/R3.2 全部完成后，§4 剩 R3.3–R3.6。13 个 `test/e2e_*.py` 断言型脚本 100% 靠人工手跑；
`make e2e` 现状是「假 e2e」——`cd test && python -m pytest -q` 因 pytest 默认只发现 `test_*.py`，
实际收集不到任何 `e2e_*.py`，静默空跑（`ci.yml` 自己的注释也承认这一点，把断言 e2e 明确推给本卡）。

## 2. 关键发现：不能把 roadmap 卡片字面的 5 个脚本原样塞进无密钥 nightly

调研（两轮 Explore + 一轮 Plan 子任务 + 直接读代码验证）确认一个决定实施方式的事实：

**Mock LLM 下云端 Planner 必然把大多数请求兜底路由到 chitchat**：`MockProvider.complete()` 返回非
JSON 文本 → `planning._parse_and_validate` 解析必炸 → 走 `_fallback()` → 落到
`PLANNER_FALLBACK_AGENT`（默认 chitchat），除非目标 Agent 声明了 `route_hints`。全仓只有
`trip_planner`、`deep_research` 两个 Agent 声明了 `route_hints`；导航/媒体/信息类都没有。

这意味着卡片字面提到的 5 个脚本（central_hub/context/memory/ws/resilience）不能整份跑，其中几个
断言在逻辑上必然失败（不是偶发）：

| 脚本/用例 | 为什么必挂 |
|---|---|
| `central_hub_cases.json` 的 `cloud_navigation_single_agent`/`cloud_media_to_edge` | 导航/媒体无 route_hints，Mock 下落 chitchat，断言的 `step.agent:navigation`/`step.edge:media.play` 不会出现 |
| `e2e_context.py` 的 `ctx_battery_query_not_chitchat` | 断言「不许落 chitchat」，但 Mock 下必然落 chitchat，直接矛盾 |
| `e2e_memory.py` 链路 1（真 embedding 语义桥接） | 需真 `LLM_EMBED_API_KEY`；零字面重叠 query 在 lexical 兜底下必空 |
| `e2e_memory.py` 链路 3（chitchat 宠物召回） | 需真 `LLM_API_KEY`；`MockProvider` 只回显用户末句，宠物名不会出现在回复里 |

反过来，`e2e_trip.py`/`e2e_research.py`/`e2e_research_async.py` 虽不在卡片字面 5 个之列，但
trip-planner/deep-research 有 `route_hints` + 确定性 fallback 生成器（`_fallback_skeleton`/
`_fallback_report`，后者显式检测 `"[mock]"` 前缀），Mock 下反而稳定可靠——一并纳入比卡片字面范围
更扎实。

## 3. 方案

1. **裁剪，不是照搬**：`central_hub_assertions.py`/`e2e_context.py` 本身已支持 `--case NAME`
   （可重复）过滤，nightly 直接传参跳过上表 4 个必挂用例，不改这两个脚本的代码。
2. **`e2e_memory.py` 打最小补丁**：链路 1/3 各加一行 guard，复用链路 6
   （`check_proactive_routine`）已有的「缺依赖优雅 SKIP」惯例——检测不到对应 key 时
   `record(..., True, "SKIP：...")`，而不是让它们进入必挂名单。
3. **`make e2e`（本地）与 nightly（CI）刻意跑不同的清单**：本地开发者 `.env` 可能配了真实 key，
   `make e2e` 跑全量清单（不加 `--case`，未配置 key 时部分用例合理失败/优雅 SKIP，属已知
   Mock 限制不是回归）；nightly 追求「零密钥、确定性全绿」，跑裁剪过的子集。两者不合并成一份，
   避免为了兼容对方互相妥协。
4. **零 GitHub secrets**：整条 nightly 不需要任何真实 LLM/三方 key——避免真实调用的不确定性拖累
   「全绿」目标，也避免碰 CLAUDE.md 红线里的「密钥」项。
5. **就绪判定**：26 个 compose 服务只有 redis/nats/postgres 三个有 healthcheck。
   `docker compose up --wait` 能等这三个健康 + 其余容器进入 running，但不保证 app 层端口真正就绪；
   额外轮询 `observability-collector:8092/healthz`（系统里仅有的两个真实健康端点之一）兜底。

## 4. 变更清单

1. **`.github/workflows/nightly-e2e.yml`（新）** — `schedule`(daily)+`workflow_dispatch` 触发；
   codegen(`buf generate proto`) → `.env`（不填 key）→
   `docker compose -f compose.yaml up --build -d --wait --wait-timeout 300` → 轮询 collector
   healthz → 顺序跑 8 脚本（`ws`/`central_hub[3 case]`/`context[4 case]`/`memory`/`resilience`/
   `trip`/`research`/`research_async`，沿用 `ci.yml` 已有的 `set +e; run_group(){...}; exit $fail`
   汇总退出码惯用法）→ 无论成败 dump `docker compose logs` 并上传 artifact → `compose down -v`。
   `timeout-minutes: 60`（26 服务全新 build 无层缓存 + `research_async` 单脚本最长 ~4.3 分钟，首次
   真跑后再按实际调整）。顶部注释消歧 `test/nightly/`（那是另一个不相关的、已有的真实 LLM 语料回归
   概念，只是碰巧同名）。
2. **`Makefile`** — `e2e:` 目标从 `cd test && python -m pytest -q` 改为 `bash scripts/run_e2e.sh`；
   不碰 `COMPOSE := docker compose -f compose.yaml` 那行（`test/test_compose_env_entrypoint.py`
   对它做字符串断言）。
3. **`scripts/run_e2e.sh` + `run_e2e.ps1`（新）** — 本地全量清单执行器：跑前探一次 collector
   healthz；顺序跑 9 个脚本（不加 `--case`）+ 最后 `python -m pytest test/e2e_real_providers.py -q
   -s`（显式路径绕开默认文件名过滤）；每步记录退出码，打印汇总表，任一失败整体非零退出。`.ps1`
   跟现有 `scripts/gen-proto.ps1` 同风格，但 `$ErrorActionPreference = "Continue"`（要跑完全部脚本
   再汇总，不能一个失败就提前退出）。
4. **`test/e2e_memory.py`** — 链路 1（`check_semantic_bridge`）/链路 3（`check_chitchat_pet`）各加
   一行 guard：未配置 `LLM_EMBED_API_KEY`/`LLM_API_KEY` 时记为 SKIP（`True` + 说明），其余逻辑不变。
5. **`test/README.md` §5`** — 补齐此前缺失的 3 个脚本文档（`e2e_auth.py`/`e2e_mtls.py`/
   `e2e_real_providers.py`）+ 一句说明 nightly 与 `make e2e` 子集刻意不同。
6. **`CLAUDE.md` §6** — Windows fallback 那句顺带补 `scripts/run_e2e.ps1`。
7. **`ci.yml`（PR 门禁）不改** — `smoke_edge` 已在跑，卡片③是确认现状不是新工作。

## 不做 / 取舍

- 不碰编排核心任何业务逻辑（`orchestrator/cloud/{planning,context,aggregator,progress,
  route_hints}.py` 零改动）。
- 不解决 D19（e2e 脚本重复的 Windows GBK 编码 hack、无共享 harness 模块）——独立技术债，本卡不顺手
  重构。
- Go/apt 国内镜像（`goproxy.cn`/`mirrors.aliyun.com`）在 GitHub Actions runner 上是否顺畅未经实测
  （`GOPROXY=...,direct` 有兜底，风险不算高）；不预先改 Dockerfile，留到首次真跑后视实际情况处理。
- `e2e_observability.py`（人工巡检，零断言，恒退出 0）与 `e2e_auth.py`/`e2e_mtls.py`（需
  `AUTH_REQUIRED=true`/`GRPC_TLS=on` 等非默认栈配置）不纳入这次的默认模式 nightly。

## 6. 首次实跑发现：链路 2 也需要 embedding-key SKIP guard（本次审查的遗漏）

**现象**：GitHub 上首次 `workflow_dispatch` 实跑，其余 7 个脚本全过，唯独 `e2e_memory.py` 汇总
5/6，`[FAIL] 2.planner召回注入 — 回复『[mock]...』，日志未见召回`。

**根因**：`check_planner_injection`（链路 2）种子记忆是「用户不吃辣，喜欢清淡」
（`predicate=taste.spicy`），探测 query 是「我想找个地方吃饭，给点建议」——两者**字面重叠很弱**，
这条测试的本意就是验证「弱字面重叠也能语义召回」（跟链路 1 的「零字面重叠」是同一类断言，只是
程度较弱）。真实 embedding 下能命中（本地长期运行的真实 key 栈上验证过：召回到
`taste.spicy`/`route.avoid_highway`/`travel.pace` 共 3 条）；但 nightly 是全新起栈、
`LLM_EMBED_API_KEY` 为空，memory 服务的召回诚实降级为 lexical，弱字面重叠在 lexical 下命中不到，
"日志未见召回"是**逻辑必然结果**，不是环境抖动或时序问题。

这是本次审查的**遗漏**：两轮 Explore + Plan 子任务调研都把链路 2 标注为"mock-safe"（因为它看
cloud-planner 日志而非 LLM 回复内容，绕开了链路 3 那种"MockProvider 只回显末句"的问题），但没有
意识到**日志里的召回记录本身也是 embedding 召回的产物**——链路 2 虽然不依赖 LLM 对话补全，但依赖
**memory 召回的语义质量**，这点和链路 1 本质相同，只是链路 1 用零重叠 query 更容易被静态分析看穿，
链路 2 用弱重叠 query 没被看穿，只有真实跑一次才暴露。

**修复**：`check_planner_injection` 加一行同款 guard（`if not os.getenv("LLM_EMBED_API_KEY")`），
与链路 1 完全对称。本地验证：`python test/e2e_memory.py`（进程自身不 export `.env`，复现 nightly
条件）→ 6/6 全过、三条 SKIP guard（链路1/2/3）均按预期触发；全量 pytest 回归零回归。已直接提交
到 main（这是对刚合并工作的直接修正，非新功能，未另开分支）。
