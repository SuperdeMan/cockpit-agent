## 1. 背景：三件事为什么合并成一个主题

三件事都作用于同一条产品能力——「用户的话被路由到对的地方」，且共享同一套评测护栏（T3.4）：

1. **T4.1/A6/D11**：Registry「语义路由」实为字符命中打分（`registry/store.py::_score`，
   `0.3+0.05×字符交集`）。规划失败降级 `planning._fallback(query, top_k=1)` 与未来 Agent 规模化后的
   catalog 预筛都靠它。
2. **K6**：端侧 `fast_intent.classify_structured` 对飞书全量 8683 例句实测识别率 **72.0%**，
   缺口分布已量化（气象 5.0%/导航 28.5%/辅助驾驶 35.9%…），扩规则 vs NLU 未决。
3. **D7**：`fast_intent.py` 1558 行规则与引擎不分离，扩覆盖会让它更大。

### 1.1 设计调研新发现：一个现存 bug（P0 顺带修复）

`registry/server.py::ResolveAgents` 已有语义检索钩子（关键词 best_score<0.5 时调
`store.resolve_semantic` 并**无相似度下限地追加**结果）；而 `PgStore._embed` 在
sentence-transformers 不存在时（**registry 镜像从未安装过它**，见 `registry/requirements.txt`
注释）回退 **sha256 hash 伪向量（384 维）**。后果：

- 今天每次 Agent 注册都在往 PG `agents.embedding` 写**语义上是随机数**的向量；
- 任何关键词低分的 query 都会触发 hash 向量 cosine 检索，把**伪随机排序的 Agent** 追加进候选；
- `planning._fallback` 取 top-1：若关键词命中为空，一个随机 Agent 会接到这句话（以空槽位执行其
  首个 capability）。低频但真实的误路由路径。

P0 的向量真化天然修复它；同时要在 server 钩子处补相似度下限（防未来任何弱向量源重现此坑）。

## 2. 目标 / 非目标

**目标**
- G1：Registry 语义检索换真向量（llm-gateway→百炼 text-embedding-v4，与 memory 同源），
  无凭证/失败时诚实降级回关键词打分（删除 hash 伪向量路径）。
- G2：把飞书 8683 例句变成**入库的覆盖率评测资产**，覆盖率报告纳入 CI（非阻塞）。
- G3：端侧识别率 72% → **≥82%**（首轮 quick-win 批次），且零劫持回归（反例护栏）。
- G4：K6 路径决策落地：**本轮走「扩规则」**；NLU 模型明确 defer，触发条件写死（见 §7）。

**非目标**
- 不上 NLU 模型/端侧分类器（本轮）；不做 fast_intent.py 整体重写（D7 完整数据化仅作可选 P4）；
- 不改编排核心与 proto；不追求 100% 覆盖（导航搜索/规划类**本就该上云**，见 §5.3 甄别）；
- 不做 Registry 灰度/多版本路由（仍属 ws2 后续）。

## 3. P0 · Registry 真语义路由（规模 S-M）

### 3.1 改动面（全部在 `registry/` 内）

| 文件 | 改动 |
|---|---|
| `registry/store.py` | ①`_embed`/`_init_embedder` 重写：经 llm-gateway gRPC `Embed`（模式照抄 `memory/pg_store.py:195`：`runtime.grpcio.aio_channel(LLM_GATEWAY_ADDR)` + `llm_pb2.EmbedRequest`，timeout 5s）；**删除 sentence-transformers 与 hash 伪向量两条路径**——embed 不可用返回 `None`（不写向量、不语义检索）。②新表 `agent_capability_vec(agent_id, intent, text_hash, embedding vector(<dim>), PRIMARY KEY(agent_id,intent))`——**按 capability 粒度向量化**（一个 Agent 一条混合向量会把 info 的天气/股票/新闻搅在一起，检索粒度必须到 capability：文本=`intent + description + examples` 拼接）。③`register()`：per-cap embed 前先比 `text_hash`（sha256），未变化跳过（**Agent 每 10s 周期重注册，绝不能每次打 embed API**）；agent deregister 级联删该表行。④`resolve_semantic` 改查新表，按 agent 聚 `max(similarity)`，返回前过 `SEMANTIC_MIN_SIM`（env，默认 0.35）；query 向量加 LRU cache（128 条 / TTL 5min）+ 1.5s 超时，失败返回 `[]`（关键词结果原样生效）。⑤旧 `agents.embedding` 列停写停读（保留列不迁移——数据可由周期重注册 10 秒内再生，无迁移成本）。 |
| `registry/server.py` | 语义追加处补 `SEMANTIC_MIN_SIM` 下限过滤（双保险，修 §1.1 bug 的另一半） |
| `registry/requirements.txt` | 删 sentence-transformers 注释段，注明 embed 经 llm-gateway |
| `deploy/docker-compose.yaml` | registry 环境补 `LLM_EMBED_DIMENSIONS`（沿用全局默认 1024，x-python-env 已有 `LLM_GATEWAY_ADDR`） |

维度：读 `LLM_EMBED_DIMENSIONS`（默认 1024，与 memory 一致）；建表用该维度，启动时若表维度不符
**直接 DROP 重建**（向量由重注册自动回填，这是本仓独有的零成本迁移优势——设计上明确利用它）。

### 3.2 行为边界（必须保持）

- 无 `LLM_EMBED_API_KEY` / llm-gateway 不可达 / PG 不可达：**行为与今天关键词路径完全一致**
  （语义分支静默缺席）——nightly（纯 mock 无 embed key）必须零感知。
- intent 精确命中仍 1.0 快路径（`_score` 不动）；语义仅在钩子条件（best_score<0.5）触发，
  不增加正常规划路径延迟。

### 3.3 P0 验收

```bash
python -m pytest registry/ --import-mode=importlib -q     # 新增单测：mock embed 的 per-cap 写入/去重/下限过滤/降级
python -m pytest --import-mode=importlib -q               # 全量零回归（≥1046）
make up && python test/e2e_ws.py                          # 真栈非破坏
# 真栈语义探针（有 LLM_EMBED_API_KEY 时）：
#   grpcurl/脚本调 ResolveAgents(query="茅台股价怎么样", top_k=1) → top-1 = info
#   ResolveAgents(query="帮我找个川菜馆订位")            → top-1 = food-ordering
# 无 key 模式：同 query 行为与改动前 byte-level 一致（日志无 semantic 分支）
```

## 5. P2 · 语料资产化 + 覆盖率报告（规模 M）

### 5.1 语料入库

- 按 gap-analysis §2 已验证的 lark-cli 命令重拉 1465 行（8 页 × 200），本地转换为
  `test/eval_corpus/feishu_intents_full.jsonl`（每行：`{text, domain, object, edge_expected}`；
  「标准说法+高频说法」拆行去重，预计 8683 行 / ~1MB，纯派生评测资产、与已入库的
  entities.yaml 同性质）。转换脚本入 `scripts/gen_intent_corpus.py`（可重跑，幂等）。
- `edge_expected` 三态：`true`（端侧应接住）/ `false`（cloud-by-design，不计入端侧缺口）/
  `null`（未甄别）。初始全 `null`，§5.3 甄别后逐域回填。

### 5.2 覆盖率报告模式

- `test/eval_fast_intent.py` 增 `--corpus full`：对全量 jsonl 跑 `classify_structured`，输出
  总体/分域/分对象识别率 + 与上次快照的 diff，产物 `docs/reviews/eval/coverage_fast_intent.{json,md}`。
  首版快照必须复现 gap-analysis 的 **72.0%**（±0.5pt，作为转换正确性的自校验）。
- CI：`intent-eval-baseline` job 追加该模式，**总体覆盖率较快照下降 >1pt 则 `::warning::`**
  （报告型，不阻塞）；39 条策展基线仍是逐例回归、性质不变。

### 5.3 导航/取消类甄别（先甄别再扩，防止两类错误）

- **导航（643 条，28.5%）**：逐句打 `edge_expected`——播报开关/音量/静音类=true（端侧车机控制）；
  搜 POI/路线规划/去某地=false（**本就该上云走 navigation Agent，不是缺口**）。甄别后产出
  第二指标「端侧应接子集覆盖率」，这才是 P3 的真实靶子。
- **交互-裸「取消」（26 条，0%）——明确不做端侧规则**：裸「取消」必须继续上云。engine 的
  F1 确认闭环靠云端 `_confirm_reply` 处理挂起任务的取消；若端侧新增规则接住裸「取消」秒回，
  会**劫持云端待确认会话的语音取消路径**（用户语音说「取消」再也到不了云端 pending session）。
  该 26 条在语料里标 `edge_expected: false` + 注释原因。**这是本卡最容易踩的坑，执行者必读。**

| 批次 | 内容 | 预期收益（语料句数） | 反例护栏（必须同批入 eval） |

3. `python -m pytest orchestrator/edge --import-mode=importlib -q` + `python test/smoke_edge.py` 13/13；
4. 全量 pytest 零回归；本批反例全绿。

## 7. K6 路径决策（本设计一并定案）

**走路径 1（扩规则）**，理由：缺口分布长尾集中在少数模式族（§6 三批可覆盖大半）、评测护栏已就绪、
架构零风险；**NLU 模型（路径 2）defer**，触发条件二选一即启动重评估：
① 两轮扩规则后总体覆盖率仍 <85%；② fast_intent 模式表规模突破 ~300 条且单批回归开始互扰
（出现一批 >2 个反例失守）。届时语料（本卡 P2 资产）即训练/评测集，前置投资不浪费。

## 8. 风险与回滚

| 风险 | 缓解 |
|---|---|
| embed API 失败/限流打在注册路径 | text_hash 去重后稳态调用≈0（仅 manifest 变更时）；失败→None→关键词路径，不阻注册 |
| 语义检索吸走本该端侧/精确命中的 query | 语义仅在 best_score<0.5 钩子触发 + SEMANTIC_MIN_SIM 下限 + P1 反例基线 |
| 新 pattern 劫持既有意图（历史上发生过：裸「电池」劫持调研） | 每批「先反例后规则」纪律 + 39 基线 + 88 corpus + 全量回归；出劫持即整批回滚（批次独立合并） |
| PG 向量表重建瞬时空窗 | 周期重注册 10s 内回填；空窗期间语义分支返回空=关键词兜底，无用户可见影响 |
| 语料含公版表衍生内容 | 与已入库 entities.yaml/responses.yaml 同源同性质（内部评测用途）；只入派生字段，不入原表全字段 |
