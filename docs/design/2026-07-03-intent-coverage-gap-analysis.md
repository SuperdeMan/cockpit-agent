## 5. 典型漏判样本（挑量级有意义的对象）

- **气象（218 条，5.0%）**：`气象查询澳门台北` / `长沙的气象怎么样` —— 语义等价的"天气"类
  （天气/空气/温度/湿度/指数）识别率都在 85%+，唯独"气象"这个同义词没进规则，**像是命名遗漏，
  修复成本低**。
- **导航（643 条，28.5%，全表最大单类）**：`打开导航播报` / `请将导航信息播报打开` —— 部分
  漏判是导航播报开关等**确实该端侧处理**的控制类意图；但导航大类本身包含大量搜索/规划类语句，
  **这部分本就该走云端 navigation Agent**，低识别率不能直接等同于"缺口"，需要先甄别子类别再
  下结论。
- **辅助驾驶（482 条，35.9%）**：`开辅助驾驶设置页面` / `关闭辅助驾驶设置界面` —— 多是
  "打开XX设置页面"这类引导类意图，和界面（150,32.7%）/语音助手（150,44.0%）/声音设置/系统
  设置等同属**"开XX设置页面"模式未被规则覆盖**这一类问题。
- **交互——裸"取消"（26 条，0.0%）**：`取消` / `取消一下` / `取消试试看` —— 完全空白，
  但语义单一、修复成本很低。
- **工具/个人中心/设置/空气净化（各 ≤24 条，0.0%）**：量小，工具类是"取力器控制"（商用车
  专属功能，`docs/design/2026-06-13-...md` 里"重汽航天"等项目相关，非乘用车通用）。

## 6. 后续方向（本轮不做决定）

三种可能路径，规模和风险差异大，需要用户决定走哪条（或都不走，只是先有这份数据）：

1. **扩规则覆盖**（M～L）：把飞书全量语料里未覆盖的高频说法补进 `fast_intent.py`
   if/elif 规则或做成数据驱动的模板生成，架构不变、风险低。可先挑「气象」「交互-取消」
   「空气净化」这类小成本高回报的做，「辅助驾驶」「空调模式」这类量大的再排期，「导航」
   需要先甄别哪些子类该留给云端。
2. **真上 NLU 模型**（L～XL）：字面对应 §8 P2 原始方向——训练/接入轻量分类器替换规则引擎，
   语料就是这 8683 条真实说法+词库实体。工作量大、需要评估端侧离线可用性和延迟，架构性改动。
3. **保持现状，此报告只作参考**：72% 整体识别率是否"够用"取决于产品目标；不动手也是一个
   选项。

## 7. 已知限制

- `+data-query` 聚合接口对此 Base 返回 `91403`（user/bot 两种身份均如此），未能用云端 groupby
  验证本文件的本地聚合结果——本地统计基于全部 1465 行的 `+record-list` 原始拉取，逻辑简单
  （字符串 split + 计数），但未经第二套独立方法交叉验证。
- 「识别率」衡量的是 `classify_structured()` 是否返回非 `None`（即"有没有被端侧接住"），
  不是"分类是否完全正确"——本文件不判断误分类（分错对象/操作），只判断漏判（完全没接住）。
- 本文件不含 §6 三条路径的实际选择与实施——留待下一轮明确方向后另立设计文档执行。
