## 1. 现状与差距

**现状链路**（伪流式，句级批处理）：
- 云端 `speech_delta`/`final` → HMI `App.tsx:128/257` → `audio.ts` 的
  `startTTSReply/appendTTSDelta/finishTTSReply` → `TtsTextBuffer` 按标点切句（≤48 字）→
  `OrderedPlaybackQueue` 并行预取 + 顺序播放 → 每句 `POST /api/tts`（llm-gateway
  `http_server.py:117`）→ `MiMoTTSProvider.synthesize`（整句 wav/pcm16 base64，httpx 60s）→
  HMI `HTMLAudioElement` 播 blob。
- `stopTTS()`（audio.ts:245）已存在：发送新文本时（App.tsx:115）会停播 + 清队列。

**差距**：
1. **首音延迟 = 首句全量合成 RTT**（典型 1.5-4s）——句子越长越慢，无法边合成边出声；
2. **无 barge-in**：TTS 播报中按麦克风（光球）不会停播报；更没有语音打断；
3. 长回复逐句请求 N 次 HTTP，无法利用增量文本（`speech_delta` 到了却要攒满一句才发）。

## 2. 目标 / 非目标

**目标**
- G1：真·流式合成——文本增量进、音频分片出，**典型句首音 <1s**（对照：批处理路径首音基线
  在 P0 探针中实测记录）。
- G2：Barge-in v1（确定性）：播报中按光球/发新消息 → 立即静音 + 取消在途合成（端到端取消，
  不浪费上游 token）。
- G3：Barge-in v2（语音打断，实验性、默认关）：播报中检测到用户开口 → 自动停播并转为聆听。
- G4：全链路可回退——流式不可用时**无感回退**现有句级批处理（复用 ASR 的降级惯例），
  nightly/mock 模式零破坏。

**非目标**
- 不做唤醒词/全双工对话；不改 TTS 批处理路径（保留为回退）；不动 proto/编排核心/网关；
- 不做多音色克隆/SSML 精细控制（沿用现有 voice 设置）；
- v2 语音打断不追求量产级 AEC 效果（浏览器 AEC 能力边界如实记录，见 §6 风险）。

## 3. 方案

### 3.1 P0 前置探针（硬 gate，半天，先于一切集成代码）

**供应商选型：DashScope CosyVoice 流式 TTS**，理由：与 fun-asr-realtime 同一
`/api-ws/v1/inference` run-task 协议壳（**该协议 2026-06-30 已破解**：run-task/continue-task/
finish-task JSON 帧 + 二进制帧，见 `llm-gateway/providers.py::DashScopeInferenceASRProvider`），
同一 DASHSCOPE key，协议知识/超时坑全部复用。MiMo TTS 走 chat/completions 整段返回，无流式接口。

探针脚本 `scripts/diag_tts_stream.py`（对标 ASR 落地时的 diag 先例）验证四件事，**全过才进 P1**：
1. 账号对 `cosyvoice-v2`（或 `cosyvoice-v1`）的 run-task 是否 200 开任务（学 qwen3 教训：
   model id 大小写/开通状态先验证）；
2. `continue-task` 增量喂文本 → 服务端**边收边回二进制音频帧**（确认不是攒到 finish 才回）；
3. 输出格式协商：优先 `pcm`（22050/24000Hz s16le），记录实际采样率（HMI 播放器要用）；
4. 测量：首帧延迟（run-task→首个二进制帧）与整句延迟，写进本文件 §7 基线表。

**探针失败的退路**（按序）：a) 换 `sambert-zhichu-v1` 等 sambert 系（同 DashScope，流式 TTS
老接口）；b) 都不可用 → 本卡降级为「P2 barge-in（对批处理路径）+ 批处理并行预取调优」，
流式合成留待供应商就绪——**探针结论写回本文件后再决定，不带着未验证假设写集成代码**。

### 3.2 P1 llm-gateway：流式 Provider + WS 端点（M）

| 文件 | 改动 |
|---|---|
| `llm-gateway/providers.py` | 新 `BaseStreamingTTSProvider`（`async def stream(text_deltas, *, voice, fmt) -> AsyncIterator[bytes | dict]`）+ `DashScopeCosyVoiceProvider`（run-task 协议，文本增量经 continue-task 转发，二进制帧原样 yield，dict 帧作控制事件）+ `MockStreamingTTSProvider`（yield 固定静音 PCM 片，**nightly/无 key 可跑**）+ `build_tts_stream_provider()` 工厂（env `TTS_STREAM_PROVIDER=dashscope|mock|off`，默认 `off`） |
| `llm-gateway/http_server.py` | 新 WS `GET /api/tts/stream`（镜像 `/api/asr/stream` 实现模式）：客户端 JSON 帧 `{type:"text", delta}` / `{type:"finish"}` / `{type:"cancel"}` 上行；服务端二进制音频帧 + JSON `{type:"meta", sample_rate, format}`（首帧前）/`{type:"done", chunks, first_chunk_ms}` 下行。cancel/连接断开 → 立即 finish/断开上游任务（**取消传播到供应商**）。另加 `GET /api/tts/stream/info`（能力探测，镜像 asr/stream/info——HMI 据此决定走流式还是回退） |
| `deploy/docker-compose.yaml` + `.env.example` | `TTS_STREAM_PROVIDER`/`TTS_STREAM_MODEL`（默认 `cosyvoice-v2`）/`TTS_STREAM_VOICE`/`DASHSCOPE_ASR_KEY` 复用（key 同 ASR） |

单测（`llm-gateway/tests/test_tts_stream.py`）：协议帧构造/状态机（run-task→continue→finish）、
mock provider 分片产出、cancel 中断、工厂路由——全部离线可跑。

### 3.3 P2 HMI：流式播放路径（M）

| 文件 | 改动 |
|---|---|
| `hmi/src/pcmPlayer.mjs`（新） | **纯逻辑可 node 测**的 PCM 调度器：chunk 入队 → 计算各 AudioBuffer 的 `start(when)` 时刻（首块攒 ~300ms jitter buffer 起播、后续无缝拼接、underrun 时重建起点）。Web Audio 对象经注入（node 测试传 fake） |
| `hmi/src/audio.ts` | 新流式路径 `streamTTSReply`：`startTTSReply` 时若「设置开 + `/api/tts/stream/info` 探测可用」→ 开一条 WS，`appendTTSDelta` 直接透传 delta（**不再攒句**），`finishTTSReply` 发 finish；收 meta 后用 AudioContext+pcmPlayer 播分片。**任一环节失败（WS 拒/超时/中途断）→ 无感回退现有句级批处理**（把已累计文本交回 `TtsTextBuffer`，惯例同 ASR 流式失败回退批处理）。`stopTTS()` 扩展：关 WS（发 cancel）+ pcmPlayer 立即静音 |
| `hmi/src/components/SettingsPanel.tsx` + `settings.tsx` | 「语音播报」区加流式开关（默认 auto：探测到即用）；沿用现有 voiceId 来源 |
| `hmi/src/pcmPlayer.test.mjs`（新） | 调度纯逻辑 node 测：起播阈值/拼接时刻/underrun 重建/cancel 清零 |

注意：AudioContext 需用户手势解锁（autoplay 策略）——HMI 每轮回复都由用户交互触发，沿用现有
`HTMLAudioElement` 能播的前提即可，但要在首次 resume 处显式 `ctx.resume()` 防挂起态。

### 3.4 P3 Barge-in（S + M 两档）

**v1 确定性打断（S，必做）**：
- `Composer.tsx` 光球按下（进入录音，无论批处理或流式 ASR）→ 先调 `stopTTS()`（当前只有发送
  新文本会停，按麦不停——补上这条）；
- `stopTTS()` 的取消传播链验收：HMI 停播 ≤100ms + WS cancel 帧 → llm-gateway 断上游任务
  （日志可见 finish/close，不再继续吐帧）。

**v2 语音打断（M，实验性，settings 默认关）**
> ⚠️ **2026-07-04 移交**：v2 已由 [`2026-07-04-r4.3-wake-vad-fullduplex.md`](2026-07-04-r4.3-wake-vad-fullduplex.md)
> P3（VAD 判据，比 ASR-partial 更早更稳且不耗 ASR 流量）**取代**。执行本卡时 v2 直接跳过，
> 只做 v1；以下 v2 原文保留仅供 R4.3 对照回声护栏思路。
- 开关开启时：TTS 播放期间保持 `StreamingRecognizer` 常开（`getUserMedia` 显式
  `echoCancellation: true, noiseSuppression: true`）；收到**非空 partial ≥2 个汉字**即判打断
  → `stopTTS()` + 该次识别继续作为新一轮输入（不丢已说的半句）；
- 自触发（回声把播报文本识别回来）防御：①浏览器 AEC 为主；②partial 文本与当前正在播的 TTS
  文本做前缀/包含比对，命中则忽略（简易回声指纹）；③连续 2 次自触发自动关闭本开关并 toast 提示。
- **CI 无法验证 AEC 效果**：fake-mic e2e 只验「打断逻辑链」（注入 partial → 停播）；真实回声
  表现走人工验收单（外放 vs 耳机两种设备各测 3 句），结论如实写回本文件——效果不达标就让
  开关继续默认关，不硬拉。

### 3.5 观测（S，随 P2 顺带）

- llm-gateway 对每次流式合成 best-effort 发 `obs.metric`：`tts_first_chunk_ms`/`tts_total_ms`/
  `tts_cancelled`（复用既有 NATS emitter 惯例）；collector `/metrics` 自动聚合（R3.6 机制），
  Dashboard 不强制加面板。

## 5. 兼容与回退矩阵

| 场景 | 行为 |
|---|---|
| `TTS_STREAM_PROVIDER=off`（默认） | 与今天完全一致（句级批处理），HMI 探测 info 404/off → 不走流式 |
| 无 DASHSCOPE key | 工厂回 mock（显式配 mock 时）或 off；绝不半启用 |
| WS 中途断 / 供应商超时 | HMI 把未播文本回灌批处理路径续播；本轮不再尝试流式 |
| nightly（纯 mock） | mock 流式 provider 出静音分片，e2e_tts_stream 断言帧序而非音质 |
| mTLS / auth 开关 | 不相关——本链路是 llm-gateway HTTP(50059)，不在 gRPC mesh / WS token 面上（与 ASR 流式同面） |

## 6. 风险

| 风险 | 缓解 |
|---|---|
| 账号未开通 cosyvoice / 协议与 fun-asr 有出入 | P0 探针硬 gate + sambert 退路 + 降级路线（§3.1），杜绝带假设集成 |

| AudioContext 采样率/重采样失真 | meta 帧显式带 sample_rate，pcmPlayer 用 `AudioContext({sampleRate})` 或离线重采样；探针阶段确认供应商实际输出 |
| 长回复 WS 与 90s 端到端窗口叠加 | TTS WS 独立于对话 WS，互不占用；cancel 传播保证不悬挂 |
| 流式路径引入新依赖 | 无新 Python 依赖（websockets/aiohttp 已有）；HMI 零新依赖（Web Audio 原生） |
