## 1. 背景（为什么做）

`docs/architecture/cockpit-agent-architecture.md` §3.3「降级策略矩阵」定义了 4 行预期降级行为：断网、云 Planner 故障、单 Agent 故障、LLM 超时。此前只有 `test/e2e_resilience.py` 测"依赖换 IP 后自愈"（连接韧性），这 4 行本身从未被自动化断言过。目标：新增 `test/e2e_degrade.py` 刻画这 4 行的**真实**当前行为，纳入 nightly（`.github/workflows/nightly-e2e.yml`，T3.3 已落地）。

## 2. 关键发现：调研中挖出 3 处真实缺口，用户决定本卡只测不修

两轮 Explore + 一轮 Plan 子任务调研（读 `orchestrator/edge/server.py`/`cloud_client.py`、`gateway/cloud/main.go`、`orchestrator/cloud/{dispatch,executor,aggregator,circuit}.py`、`llm-gateway/{server,providers}.py`）发现 3 处非本次改动引入的真实正确性问题：

1. **`executor.py::_to_result()`（135-170 行）丢弃 `resp.error.message`/`.code`**，导致 `aggregator.py` 已写好的 `_ERROR_FRIENDLY` 友好话术映射（`circuit_open`→"该服务暂时不可用"等）对大多数快速失败（非超时）是**死代码**——用户实际看到的是通用「抱歉，处理失败。」，不是设计意图里更具体的提示。
2. **`gateway/cloud/main.go`（328-339 行）cloud-planner 建流后中途崩溃时只打日志、不回任何消息**——区别于"一开始就不可达"（306-326 行，有明确"云端处理异常"提示 + 自动重连重试一次）。这条无消息分支会让请求静默挂到 ~90s context 超时，边侧无任何逐请求超时兜底。
3. **`llm-gateway/server.py::CompleteStream()`（104-124 行）没有 `Complete()`（74-102 行）那样的备用模型重试**——流式请求一旦失败直接中止，不会像非流式那样试 `LLM_MODEL_FALLBACK`。

**已征询用户意见并确认**：本卡只刻画现状（哪怕现状不完美），不顺手修这 3 处——避免"测试卡"变成"顺手改核心逻辑卡"，且 3 处都不在 T3.5 验收范围内。写进本文档留后续（可能开单独任务修，不追踪具体 ticket）。

## 3. 方案：四行怎么测、为什么这么测

**注入手段选型**（区分"换不换 IP"）：

| 行 | 注入命令 | 换 IP？ | 恢复预算 |
|---|---|---|---|
| 单 Agent 故障 | `docker compose stop/start trip-planner-agent` | 否 | 60s |
| LLM 超时 | `docker compose up -d --force-recreate --no-deps llm-gateway`（注入 `LLM_MOCK_DELAY_MS`） | **是**（四行唯一） | 120s |
| 云 Planner 故障 | `docker compose stop/start cloud-planner` | 否 | 60s |
| 断网 | `docker compose pause/unpause cloud-gateway` | 否 | 60s |

- **单 Agent 故障选 trip-planner，不选 navigation 等**：mock 模式下没有 `route_hints` 的 Agent 请求会被兜底路由到 chitchat（T3.3 已确认的事实）——测"单 Agent 故障"必须选一个有 `route_hints` 的 Agent（trip_planner/deep_research）才能保证路由本身确定，否则测的是"chitchat 好好的"而非"故障 Agent 被正确处理"。注意 compose 服务名 `trip-planner-agent` ≠ span 里的 agent_id `trip-planner`。
- **单 Agent 故障断言可观测 span 而非聚合器话术原文**：因缺口①，快速失败的聚合话术是通用「抱歉，处理失败。」——断言这句等于把已知缺口的表现当"正确行为"钉进测试。改断言 `step.agent:trip-planner` 的 trace span `status=="err"`（复用 `e2e_central_hub_assertions.py` 已有 `_span_status`/`_wait_trace`），更精确、不随话术措辞漂移，也不需要先修缺口①才能写出有意义的断言。
- **LLM 超时是唯一需要加代码的地方**：现状没有任何机制能让 LLM 变慢（`MockProvider` 秒回，无延迟旋钮）。加 `LLM_MOCK_DELAY_MS`（`llm-gateway/providers.py::MockProvider.complete()`，默认 `"0"` 零行为变化）——这是**纯新增测试钩子**，不是修 3 处缺口中的任何一个。用 chitchat 做探针（`latency_budget_ms=2500ms`，比通用默认 5000ms 更快触发外层 `executor.py` 的 `asyncio.wait_for` 超时），注入延迟 8000ms 确保稳定超过。该路径触发的 `_ERROR_FRIENDLY["step_timeout"]` 是三处缺口都不影响的一条，断言固定字符串「抱歉，处理超时了，请稍后再试。」。
- **云 Planner 故障 / 断网 断言真实话术**（都已核对到具体代码行，不是照抄架构文档的措辞）：
  - Planner 故障：`"云端处理异常，请稍后重试。"`（`gateway/cloud/main.go:320`，检测到 `Unavailable` 重连重试一次后仍失败时发出）。必须在发请求**前**就停掉 cloud-planner，避免竞态踩上缺口②那条静默挂起分支。
  - 断网：车控（"打开空调26度"）走本地 `fast_intent`/VAL，完全不碰 cloud-gateway，应秒回；复杂请求经 `cloud_client.py` 的 `_CONNECT_WAIT_S`(10s) 超时后，`orchestrator/edge/server.py:527` 产出 `"网络不太好，复杂请求暂时无法处理，不过车内控制依然可以正常使用。"`（注意不是架构文档写的"网络不可用"）。
- **LLM 超时那行换 IP，需要更长恢复预算 + 限制探测次数**：注入 env 只能靠 `--force-recreate`，下游 chitchat-agent→llm-gateway 的 channel 需重连；反复探测/轮询可能在重连窗口内误触发 `dispatch.py` 熔断器（`CIRCUIT_FAILURE_THRESHOLD=5`）——检测阶段限 3 次尝试（远低于阈值）+ 恢复阶段给足 120s（对齐 `e2e_resilience.py` 同类换 IP 场景的 `RECOVER_DEADLINE`，分析确认即便偶发触发熔断，30s `recovery_timeout` 后半开探测也能在预算内收敛）。其余三行不换 IP，60s 足够。
- **不能直接复用 `e2e_central_hub_assertions.py::_send()`**：它收到第一条 final 后还会再等 10s recv 超时才返回，会把"车控秒回"这类计时断言的下限拖到 10s。改用从 `e2e_resilience.py` 复制的 `ask()`（收到第一条 final/error 立即返回）；只有单 Agent 故障那行因需要 `_wait_trace` 按 `trace_id` 查 span，用 `_send()`（它会注入 `meta.trace_id`）。
- **顺序与隔离纪律**：4 个用例严格顺序执行（不并发，爆炸半径有重叠）；每个用例的故障注入包在 `try/finally` 里，恢复后二次轮询确认真的恢复（不是发个命令就当完事）；按爆炸半径从小到大排（单 Agent → LLM 超时 → Planner 故障 → 断网）；整个 `e2e_degrade.py` 排在 nightly 和本地清单的**最后一个**——万一某次恢复没做干净，也不会连累同批跑的其它 e2e 脚本（teardown 本来就会无条件收尾整个栈）。

## 6. 真实跑暴露的问题（设计假设 vs 实测，两处推翻重来）

代码写完静态审阅没问题，但**真实对着本地全栈跑，头两轮都不是 4/4**——这里如实记录发现
了什么、为什么改，而不是把最终版本包装成"一次到位"。

**① Row 4（LLM 超时）原设计彻底不成立，改用更朴素但真实的断言。**
原计划：用 chitchat（`latency_budget_ms=2500`）+ 8000ms 注入延迟，断言命中
`aggregator._ERROR_FRIENDLY["step_timeout"]` 的固定话术。真实跑发现 chitchat 完全不受
executor 层 `asyncio.wait_for(latency_budget_ms)` 管辖——它走 `engine.py` 的 **D0 单步
流式直通**（AGENTS.md 早有记录："engine 单步 ExecuteStream 直通 + chitchat 快模型/兜底"，
本次是第一次因为要卡超时而撞上它的后果）：8000ms 延迟下请求仍在 ~16s（≈规划两次重试各
计一次延迟）内收到完整、连贯的回复，从未降级。换成唯一 mock 下路由确定的 trip-planner
（`latency_budget_ms=40000`）试 45000ms 延迟，"heavy"任务的实际时间预算（为容纳"思考"
被放宽，见 `docs/design/2026-06-24-complex-task-thinking-and-process-region.md`）比预期
大得多——耐心等到 200s 仍未收尾，作为 e2e 用例完全不现实。
**结论**：在这套系统的当前路由/执行架构下，"干净地在几秒内触发 `step_timeout` 友好话术"
本身就没有一条实际可达、够快的路径——mock 下唯二能确定性路由的两个 Agent 都是"heavy"、
时间预算被特意放宽；不需要 route_hints 的 chitchat 又走了绕开超时包装的直通路径。这是一
个有意思的架构发现，值得记录，但不在本卡修复范围。**改法**：Row 4 保留
`LLM_MOCK_DELAY_MS` 钩子（证明它本身生效），但把断言换成更朴素、真实成立的性质——LLM
变慢时系统**仍保持响应、给出连贯回复、不挂起不崩溃**（断言耗时下限证明延迟真的生效 +
仍收到非空 `final`），不再断言那句在当前架构下走不到的精确超时话术。

**② Row 1（断网）意外发现第 4 处真实缺口：pause/unpause 后 edge-orchestrator 不会自愈。**
真实跑 `docker compose pause/unpause cloud-gateway` 后，`edge-orchestrator` 的持久 bidi
channel（R2.3 落地）没有像 `e2e_resilience.py` 测的"`--force-recreate` 换 IP"场景那样
可靠重连——容器日志反复出现 `Missed too many pongs, forcing reconnect` 后紧跟着
`Cloud unavailable, degrade: cloud channel not connected`，手动等待数分钟仍未自愈，最终
只有 `docker compose restart edge-orchestrator` 能让连接恢复。这是**同 IP 冻结再解冻**
这一失效模式特有的（不同于换 IP 场景，那个场景的 keepalive+DNS 重解析机制已被验证有效），
本次审查独立发现、非本卡改动引入，且不在本卡修复范围内（会涉及
`orchestrator/edge/cloud_client.py` 重连逻辑，属编排/运行时核心代码）。
**改法**：`case_network_outage` 的 `finally` 恢复步骤不能依赖这条不可靠的自愈——
`unpause` 之后显式追加 `docker compose restart edge-orchestrator` 作为确定性兜底，确保
测试结束时栈真的恢复干净，不把坏状态遗留给后续脚本或下次本地使用。

**③（次要）Row 3 恢复探针本身偏慢，非缺陷。** `trip-planner-agent` `stop`/`start` 后进程
本身数秒内就绪，但本地真实 key 环境下用完整 `trip.plan` 请求做恢复探针，真实 LLM+高德
POI 生成完整行程可能要 30-60s+——跟"有没有恢复"无关，是这个操作本来就慢（nightly mock
模式下 trip-planner 自身的 `_fallback_skeleton` 兜底生成器应该快得多）。退避窗口放宽到
10+20+30+40=100s，仍可能在本地真实环境下显示"未确认"，但这只是 `finally` 块的诊断信息，
不影响用例本身通过/失败——已在代码注释里说明，不继续追这个真实环境特有的慢。
