## 1. 背景（为什么做）

审计指出权限模型**双轨、实为三处实现**，且文档与实现不符：

- **死轨**：`security/permission.py::PermissionEngine`（`effective_scopes`/`check`）被注入
  `PlannerEngine.__init__`，但**生产从不调用**（仅测试）；`engine.py::_enforce_permissions`
  是空壳（只 log、原样返回 plan）。
- **活轨（真正生效，逻辑重复两处）**：
  1. 规划期 `planning.py::_filter_by_permission`（fail-closed，把越权 Agent 从 catalog
     剔除，不暴露给 LLM）；
  2. 执行期 `dispatch.py::dispatch`（每步 `is_scope_covered` + `third_party`/`tool`
     硬禁令 → REJECTED）。
- **fail-open**：`context.py::_POC_DEFAULT_SCOPES`——请求不带 `granted_scopes` 时默认全授
  （有 warning，无开关）。
- `docs/architecture/detailed/ws8-security-permission.md` 声称"统一 PermissionEngine、
  规划/执行双层校验"→ 与实现不符。

**目标**：把权限**决策**收敛为唯一实现，两个运行时调用点复用它；删死壳；fail-open 加
env 开关 + 结构化审计；文档改为如实。消除"改一处漏一处"。

## 2. 关键设计判断：对审计原话的一处纠偏

审计"推荐**接线 PermissionEngine**（`effective_scopes`）"经核对代码**会真实回归**：
`effective_scopes` 用 `cap & granted` **扁平集合交集**，不做父子覆盖。而
`_POC_DEFAULT_SCOPES` 授的是父 scope `vehicle.control`，first_party cap 里只有子 scope
（`vehicle.control.hvac/.window/.seat`）——交集丢掉父 scope →
**`scene-orchestrator`（requires `vehicle.control`）会被误拒**；`dispatch` 现有测试
`test_parent_permission_covers_child_scope`（`trust_level=""` + granted 父 scope）同理会挂。

**故采用 Option A：零行为变化单轨**——统一到运行时**已验证**的 `is_scope_covered`
（父子覆盖）+ `third_party`/`tool` 硬禁令模型，`PermissionEngine.check` 委托它；trust-cap
强上限（需把 caps 改父子感知）推迟到 **R3.1**（真实 token scope 落地时；当前无 Agent
需要超 cap 的 scope，运行时强制 cap 现实收益≈0）。仍满足审计验收"全仓只剩一处权限决策实现"。

## 3. 变更清单（全部零行为变化，除显式标注）

1. **`security/permission.py`** — 新增模块级唯一决策 `check_permission(*, agent_id,
   trust_level, required, granted, kind="agent") -> Decision`（逐字复刻现行
   dispatch/planning 决策）；`PermissionEngine.check` 改为委托它；`effective_scopes`
   保留并标注为 R3.1 目标态原语（不在决策主链）；删已无调用者的 `_explain`。
2. **`security/__init__.py`** — 追加导出 `check_permission`。
3. **`orchestrator/cloud/planning.py::_filter_by_permission`** — 保留 `granted is None`
   短路；改调 `check_permission`（`kind="agent"`）；删内联重复实现。
4. **`orchestrator/cloud/dispatch.py::dispatch`** — 开头三处内联权限块 → 一次
   `check_permission` 调用；审计/metrics/REJECTED 语义不变；熔断/传输不动。
5. **`orchestrator/cloud/engine.py`** — 删 `_enforce_permissions` 空壳及其调用点
   （dispatch 已按步真校验，此壳纯冗余）；`perms` 注入与构造签名不动。
6. **`orchestrator/cloud/context.py::build_context`** — fail-open 加 env 开关
   `PERMISSIONS_FAIL_OPEN`（默认 `true`=保持现状；`false`=fail-closed，无 scope 时
   `granted=[]`，仅无权限 Agent 通过）+ 记结构化审计事件。
7. **`security/audit.py`** — `AuditLogger` 加 `fail_open_scopes(...)` 结构化事件
   （复用现有 `AuditEvent`/`log`）。
8. **文档同步** — `ws8-security-permission.md` 改为如实（唯一决策=`check_permission`，
   `effective_scopes`=R3.1 目标态，fail-open 由 env 门控）；`AGENTS.md §4` +
   审计 roadmap 勾选 R2.2 ✅。
9. **测试** — 新增 dispatch 非车控越权硬拒用例；`check_permission` 表驱动契约锚点；
   `PERMISSIONS_FAIL_OPEN=false` fail-closed 用例；回归 `test_ws8_security`/
   `test_permission`/`test_dispatch`/`test_planning`/`test_regression_intent_integrity` 全绿。

## 4. 迁移纪律

- `check_permission` 输出与现行 dispatch/planning 决策**逐字等价**（reason 措辞、`missing`
  内容、`code=permission_denied`、REJECTED）——每改一处立即跑对应测试目录。
- 不改 proto、不改 `Step`/`PlanContext` 字段、不改 `PermissionEngine.__init__` 签名。
- 影响面全在编排核心（`planning`/`dispatch`/`engine`/`context`），仅收敛权限决策路径，
  不碰路由/装配/聚合语义（故按 CLAUDE.md 走 Plan Mode 后动手）。
