## 1. K1：持久通道 pause/unpause 后不自愈（真实可靠性缺口）

### 1.1 背景

`test/e2e_degrade.py` 的 Row 4（断网）在 R3.5 真实跑时发现：`docker compose pause/unpause cloud-gateway`
（同 IP 冻结再解冻）后，edge-orchestrator 的持久 bidi channel（R2.3 落地）不像 `e2e_resilience.py` 测的
"`--force-recreate` 换 IP"场景那样可靠自愈——日志反复 `Missed too many pongs, forcing reconnect` 后一路
`cloud channel not connected`，需重启 edge-orchestrator 才恢复。R3.5 当时以"显式 restart 兜底"绕过、记录留后续。

### 1.2 真正根因（比验收报告推测的更深，两条独立缺陷）

验收报告 §4 推测修法是"`_reader` 对心跳超时主动断流重建"。真实对着全栈定位，暴露出**两条**独立缺陷：

**根因 A（主因）：心跳强制重连的 `CancelledError` 把 `_run` 重连循环打死。**
冻结场景**不是**靠 grpc keepalive 检测的——真实日志里检测断连的是 `cloud_client.py` 的**应用层心跳**
（`_ping_loop`，累计丢 pong 触发 `_cancel_stream()`）。`_cancel_stream()` 调 `self._stream.cancel()`，
`_read_loop` 的 `await self._stream.read()` 随之抛 `asyncio.CancelledError`。而 `_run` 的
`except asyncio.CancelledError: raise` **把它当"任务被取消"无条件 re-raise，整个 `_run` 重连循环任务直接死亡**
——通道再不会重连，`handle()` 一路 `_connected.wait()` 超时 → "not connected"。
换 IP 场景不中招，是因为它走 grpc keepalive 抛的普通 `AioRpcError`（走 `except Exception` 正常重连）——
两条检测路径命运不同，这解释了"换 IP 自愈、冻结不自愈"的差异。

**根因 B（次因/纵深）：重连 `_open()` 无界，冻结窗口内发起的握手会永久挂起。**
即便修好根因 A，`_run` 回到 `_open()` 时若仍在冻结窗口，新连接会卡在半开的 HTTP/2 握手（服务端冻结不发
HelloAck），无 deadline 的握手 `read()` 永久挂起；部分 Docker/WSL2 环境下即便解冻也不恢复。

### 1.3 修复（`orchestrator/edge/cloud_client.py`，两处）

1. **`_run` 用 `_closing` 区分"流被取消"与"任务被取消"**：
   ```python
   except asyncio.CancelledError:
       if self._closing:      # aclose() 先置 _closing=True 再 cancel 本任务 → 真取消，退出
           raise
       # 否则是 _ping_loop 强制重连 cancel 掉 stream 引发的 → 当作会话结束，落 finally 退避重连
       logger.warning("Cloud channel stream cancelled, reconnecting")
   ```
2. **`_open()` 有界**：`await asyncio.wait_for(self._open(), timeout=_CONNECT_WAIT_S)`——任何单次(重)连
   尝试有界失败，卡死的 channel 被丢弃、退避后用**全新 channel** 再试，解冻后干净连接即可自愈。

**未改** proto / 编排核心 / 云侧 gateway（gateway 的 Connect 读循环本就对每个 Ping 回 Pong、发 HelloAck，无需动）。

### 1.4 验证

- **单测双向守护**（`orchestrator/edge/tests/test_cloud_client_reconnect.py`，2 例）：
  - `test_ping_forced_reconnect_does_not_kill_run_loop`：刻画根因 A——app 心跳缺 pong→`_cancel_stream()`
    令 `read()` 抛 `CancelledError`，断言 `_run` 重连到健康流而非被打死。去掉根因 A 修复即失败（实测捕获日志
    正是 `Missed too many pongs, forcing reconnect`）。
  - `test_reconnect_is_bounded_and_self_heals`：刻画根因 B——冻结窗口内握手有界超时后重试，恢复即自愈。
    去掉 `_open()` 超时即在第一条冻结尝试上永久挂起而失败。
  - 两条都用「前几次流冻结、之后回 HelloAck/Pong」的假流，`asyncio.run` + `stub_factory` 注入，毫秒级跑完。
- **真栈**：冻结 cloud-gateway 80s（> 检测阈值，确保冻结窗口内已发起过重连尝试）→ 解冻 → **不重启
  edge-orchestrator**，cloud 请求 **~2s 恢复健康**（冻结期正确降级"网络不太好…"，恢复后正常作答）。
- **`test/e2e_degrade.py` 的 Row 4 从"restart 兜底"改回"自愈断言"**：`case_network_outage` 现把
  "持久通道 pause/unpause 后自愈（未重启 edge-orchestrator）"纳入通过/失败判定，`finally` 仅在异常残留时
  兜底解冻。真栈实跑 `--case network_outage` → PASS。

## 2. K2：`e2e_process_region.py` 既有失败（默认泊车态断言）

### 2.1 根因

`driving` 由 Edge 按 VAL 实时 `speed_kmh`/`gear` 标注（`orchestrator/edge/server.py::_is_driving`）。
`e2e_process_region.py` 断言"默认泊车态 `driving=false`"，但**不先把车辆置为默认态**——在跨会话长期运行、
被历次调试反复设过车速/挡位的**共享栈**上，VAL 内存态残留非泊车值（如 `speed_kmh=130`/`gear=D`），断言即失败。
不是代码回归（`server.py`/`val.py` 未变），是**测试不自足**。R3.3 落地记录已诊断出这一点。

### 2.2 修复（`test/e2e_process_region.py`）

断言前经 collector `POST /api/debug/vehicle` → NATS `obs.debug.vehicle.set` 把 `speed_kmh=0`/`gear=P`
复位（与 `e2e_central_hub_assertions.py` 同款通道，stdlib `urllib`，零新依赖），并留 1s 给 NATS→VAL 传播。
**best-effort**：调试接口关（`DEBUG_VEHICLE_CONTROL`）或不可达时不硬失败——干净栈本就是泊车态。

### 2.3 验证（双向）

- 先污染车态 `speed_kmh=130`/`gear=D`：**旁路复位**跑 → 7 条过程事件全 `driving=True`，断言 `[FAIL]`（复现原失败）。
- 同样污染态下**带复位**跑 → 复位纠正为泊车态，`driving=false`，`✓ 全部通过`（证明复位非空转、真正纠正污染）。

## 3. N1：`PermissionEngine` 死注入清理

### 3.1 根因

R2.2 权限单轨化后，唯一决策点是 `security.permission.check_permission`（规划期 catalog 过滤 + dispatch
执行期硬拒同源复用）。但 `orchestrator/cloud/main.py` 仍构造 `PermissionEngine()` 注入
`PlannerEngine(perms=...)`，`engine.py` 存 `self.perms` 却全程不用——单轨清了、注入残骸没清（误导性死代码）。

### 3.2 修复（删除，而非注释保留）

**彻底删除** `perms` 注入：`main.py`（构造 + 注入 + import）、`engine.py`（构造签名 + `self.perms` + import）、
以及 8 个测试文件里 `perms=PermissionEngine()` 构造与 import。`engine.py` 留注释声明单轨接线点：trust-cap
强上限（验收报告 K4）待 scope 层次化/IdP 后**扩 `check_permission`**、不在编排层复注入权限引擎。
`PermissionEngine` 类本身保留（`security/` 及 `test_ws8_security.py` 仍用，其 `check` 委托 `check_permission`）。

**为什么删而非留**：R2.2 已确立"编排层不做权限决策"，vestigial `perms` 参数与单轨故事矛盾；真正的未来接线点是
`security/permission.py` 而非 `engine.perms`，留着是**假的接线点**（false affordance）。删除让单轨故事干净。

### 3.3 验证

`pytest orchestrator/cloud/tests security/tests test/scenarios/test_ws8_security.py` → 213 passed；全量零回归。
