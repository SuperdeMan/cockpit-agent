## 1. 背景（为什么做）

审计 D1「全链路零鉴权」是技术债清单里唯一的 **P0-#1**：WS 无鉴权、Hello 无 token、
gRPC 全 insecure、`user_id="u1"` 硬编码、`_POC_DEFAULT_SCOPES` fail-open。目标是给出**最小
闭环路径**——不追求量产级 IdP/JWT，先用静态 token 打通「HMI 带 token → 网关校验 → 身份+scope
注入 → 下游按真实 scope 决策」的链路，并全 env 门控、默认关（不破坏现有 `make up`/e2e 开发流）。

**关键前提（R2.2 已备好下游）**：`context.py::build_context` 已消费 `meta["granted_scopes"]`
且有 `PERMISSIONS_FAIL_OPEN` fail-open/closed 开关。本任务只需把**真实 scope 喂进这条已存在
的 meta 通道**，**无需改 `planning/engine/aggregator/progress` 编排核心**。Hello proto 已带
`session_token`/`device_cert_fingerprint`/`session_resume_token`/`meta`（`channel.proto:38-44`）
→ **无需改 proto、无需 codegen**。

## 2. 设计（两层鉴权，全 env 门控、默认非破坏）

沿用 R2.2/R2.3 纪律：新增 env 开关，默认保持现状，量产/演示翻转。

### 层 1 — 用户/请求鉴权（HMI ↔ edge-gateway）
- HMI WS 连接带 `?token=`（`VITE_WS_TOKEN` 注入）。
- edge-gateway upgrade **前**取 `?token=` 查静态表 `AUTH_TOKENS` → `{user_id, vehicle_id, scopes}`：
  - 命中：连接生命周期持有身份，对每条请求 stamp `Context.UserId/VehicleId` + `meta["granted_scopes"]`。
  - 未命中/缺失：`AUTH_REQUIRED=true` → 回 **HTTP 401**（握手即失败、连接不建立）；
    `false` → 匿名回退（`user_id=AUTH_DEFAULT_USER_ID` 默认 `u1`、不注入 scope），**逐字等价今天**。
  - token 有效时**无论开关**都用其身份+scope（happy path 两模式都走到）。
- **网关对 `granted_scopes` 唯一权威**（`stampScopes`）：先剔除客户端可能伪造的值，再按 token 注入
  （防匿名模式下客户端自带 `granted_scopes` 提权）。

### 层 2 — 通道/车辆鉴权（edge-orchestrator ↔ cloud-gateway）
- edge-orchestrator `cloud_client._hello_frame()` 在 Hello 带 `session_token=CLOUD_CHANNEL_TOKEN`（默认空）。
- cloud-gateway `Connect` 收 Hello 时按 `channelTokenAllowed`：`AUTH_REQUIRED=false` 恒放行；
  `true` → token 须非空且 ∈ `CLOUD_CHANNEL_TOKENS` 否则 `HelloAck{ok:false}`。

### 复用的既有机制（不重造）
- `cloud_client.handle()` 已用**首个请求的 `context.vehicle_id` 作 Hello 握手身份**，与 edge-gateway
  stamp 的 vehicle_id 一致，天然对齐 cloud-gateway `bindRequestVehicle` 的车辆一致性校验。
- 混合意图子请求 `cloud_req`（server.py）已拷贝 `meta`/`context`，stamp 的身份/scope 自动透传。
- 权限决策唯一入口 `security/permission.py::check_permission`（R2.2）——scope 到 meta 即自动生效。

## 4. 迁移纪律 / 取舍
- 全 env 门控、默认关；匿名回退逐字等价今天（`user_id=u1`、无 scope → fail-open）。
  `e2e_ws.py`（无 token）作为默认模式回归锚点。
- **不动编排核心**（`planning/engine/aggregator/progress` 零改）、**不改 proto**。
- vehicle_id 一致性：PoC 单车，token 的 vehicle_id 与 `VEHICLE_ID` 对齐；多车/多用户不在本卡范围。
- **仍是 insecure gRPC**（mTLS 是 R3.2/T3.2）；静态 token 非量产 IdP——本卡只给「最小闭环」，
  真实 IdP/JWT 轮换/设备证书是后续。

  ① 无 token 连接被拒（edge-gateway 日志 `WS rejected: missing/invalid token` + 客户端 401）；
  ② 带 token 车控出 action；③ 带 token 云端导航出结果（Hello channel token 通过）。
- **granted_scopes 来自 token 证据**：秒模式 token'd 请求在 cloud-planner **无** `fail_open_default_scopes`
  审计事件（对比默认模式每请求都有），且 `memory recall for u1`（user_id 由 token 解析）。
