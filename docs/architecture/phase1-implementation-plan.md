# Phase 1 实施计划（工程化）

> 版本：v1.0
> 日期：2026-05-29
> 依据：`cockpit-agent-architecture.md` §14 Phase 1
> 读者：研发负责人、各模块开发、测试、PM
> 目标读法：本计划以**工作流(WS)**为单位拆解，每个 WS 含任务、交付物、验收(DoD)、依赖、规模估算；末尾给里程碑排期与 Phase 1 退出标准。

---

## 0. 一页速读

- **Phase 1 目标**：在 Phase 0 可运行骨架之上，达成「全能力域真实可用 + 可插拔生态成型 + 端云通道稳定 + 安全/可观测/评测齐备」，可进入真实场景内测。
- **10 个工作流**：契约&SDK、Registry 生产化、Planner 增强、端云通道、Memory/画像、真实能力对接、语音链路、安全权限、可观测&评测、端侧工程化预备。
- **3 个里程碑**：M1 基础贯通 → M2 能力与体验 → M3 安全与生态。
- **规模**：约 12–16 周（按 6–8 人团队并行；估算需团队按真实接口就绪度校准）。
- **关键前提**：地图/POI、媒体、支付网关、ASR/TTS、车辆状态接口的对接就绪时间（见风险 R-A）。

---

## 1. 范围

### 1.1 做（In Scope）
全部 4 个 core Agent + 首批 6 个 eco Agent 接真实/准真实能力；Registry 健康探测与灰度；Planner 并行编排、多轮澄清、聚合话术、权限强校验；端云双向流式长连接与降级矩阵；Memory 画像库与合规；语音 ASR/TTS 闭环；OTel 全链路追踪；意图与端到端评测集并入 CI。

### 1.2 不做（Out of Scope，留 Phase 2）
真实车控（SOME-IP/CAN）对接与功能安全评估；端侧件 C++/Rust 化与资源达标；第三方 Agent 正式审核与商用沙箱；OTA 生产发布流程；端侧大模型微调。

### 1.3 Phase 0 已就绪（前提资产）
proto 契约、Agent SDK、Registry/LLM-Gateway/Planner/Memory/Gateway/Edge 骨架、10 个 Agent（4 core + 6 eco，mock 能力）、docker-compose、smoke/e2e 框架。

## 2. 工作流分解（WS）

> 规模标记：S≈1人周，M≈2–3人周，L≈4–6人周。所有 WS 的"验收"即该 WS 的 Definition of Done。

### WS1 · 契约 & SDK 增强 ‖ 规模 M ‖ 依赖：无
打牢全局地基，先于多数 WS。
- 任务：
  - proto：补 `request_id`/`trace_id` 透传规范、统一错误码枚举、超时与取消语义。
  - SDK：`ExecuteStream` 流式贯通到 Planner→Gateway→HMI；统一异常→`ErrorInfo` 映射；`require_confirm` 多轮状态约定。
  - 契约测试框架：`agents/_sdk/testing` 增加"manifest 一致性校验 + 黄金用例 + 流式断言"，提供 `make test` 统一入口。
- 交付物：更新后的 proto + 重新 codegen；SDK v0.2；契约测试模板。
- 验收：任一 Agent 用模板写 3 条黄金用例并在 CI 通过；流式话术能端到端逐字到达 HMI。

### WS2 · Registry 生产化 ‖ 规模 M ‖ 依赖：WS1
- 任务：心跳健康探测与失联自动摘除；注册信息持久化（PostgreSQL）；多版本并存 + 按 `vehicle_group`/比例灰度路由；`ResolveAgents` 语义路由从关键词升级为向量检索（capabilities/examples 向量化）。
- 交付物：Registry v0.2；灰度路由配置；向量路由索引。
- 验收：杀掉一个 Agent 实例后 ≤10s 自动摘除且不被路由；同一 agent_id 双版本可按比例分流；语义路由 top-1 命中率在评测集 ≥ 既定基线。

### WS3 · Cloud Planner 增强 ‖ 规模 L ‖ 依赖：WS1, WS2
编排核心，价值最高、复杂度最高。
- 任务：
  - 并行 DAG 执行（含步骤间参数依赖、超时、熔断、部分失败处理）。
  - 多轮澄清/补槽：`NEED_SLOT`/`NEED_CONFIRM` 的会话状态保持与续接。
  - 结果聚合：多 Agent 结果经 LLM 改写为连贯口语 + 结构化卡片。
  - 权限强校验：调用前按 `requires_permissions` × 授予集拦截，越权 `REJECTED`。
  - **车控 action 回流端侧**：云端产出的 `vehicle.control` 动作下发到端侧 VAL 校验执行（落实"规划/执行分离"跨端云）。
- 交付物：Planner v0.2；编排执行引擎；会话状态存储（Redis）。
- 验收：跨域组合意图（"找顺路川菜馆订位"）端到端成功；缺槽位能追问并续接；越权调用被拦截；云端车控动作经端侧 VAL 执行并可审计。

### WS4 · 端云通道 ‖ 规模 M ‖ 依赖：WS1
- 任务：Edge↔Cloud 双向流式长连接（gRPC over HTTP2/QUIC）、心跳、断线重连、指令幂等；Cloud Gateway 鉴权（设备证书 + 会话 token）；§3.3 降级矩阵全量落地（断网/云故障/Agent故障/LLM超时）。
- 交付物：稳定端云通道；鉴权中间件；降级策略实现 + 用例。
- 验收：弱网/断网注入下，车控/媒体 0 中断，复杂意图给出正确降级话术；非法设备被拒。

### WS5 · Memory & 画像 ‖ 规模 M ‖ 依赖：WS1
- 任务：`vehicle.*` scope 接车辆状态服务（实时）；`profile.*` 接画像库（结构化 + 向量）；上下文 scope 级权限；画像可查看/导出/删除（合规）；敏感 scope 端侧优先、上云最小化。
- 交付物：Memory v0.2；画像库 schema；合规接口（导出/删除）。
- 验收：Agent 按 scope 取数受权限约束；画像删除后不可再被检索；位置等敏感数据上云走脱敏路径。

### WS6 · 真实能力对接 ‖ 规模 L ‖ 依赖：WS1, WS3, WS5
逐个把 mock 换真。可在 Agent 间并行。
- navigation：接地图/POI（高德/百度/HERE 适配层）、实时路况、充电桩。
- media：接媒体内容源播放控制。
- info：天气/新闻/日程/提醒真实数据源。
- food-ordering / parking-payment：接到店/停车平台 + **统一支付网关**（Agent 不持凭证）。
- manual-rag：mock KB 换车型手册向量库（embedding + 重排 + 多车型隔离 + 出处）。
- trip-planner：升级为**跨 Agent 协作**（调 navigation/info 等），作为子规划者。
- 交付物：各 Agent v0.2（真实能力）+ 各自外部适配层 + 契约测试。
- 验收：每个 Agent 黄金用例（真实/沙箱环境）通过；支付走网关且可审计；trip-planner 能联动至少 2 个 Agent。

### WS7 · 语音链路 ‖ 规模 M ‖ 依赖：WS4
- 任务：端侧流式 ASR + 云端增强；流式 TTS 播报；唤醒与打断（barge-in）；HMI 接真实语音输入与逐字渲染。
- 交付物：语音接入服务；HMI 语音交互。
- 验收：语音→意图→执行→TTS 全链路闭环；可打断；首字 TTS 时延达标。

### WS8 · 安全与权限 ‖ 规模 M ‖ 依赖：WS3
- 任务：permission scope 强校验落地（与 WS3 协同）；`third_party` Agent 运行沙箱 + 网络出口白名单；车控安全态门控清单（行驶/速度门控）；LLM 注入防护（输入/系统指令隔离、工具参数 schema 校验）；内容审核接入 LLM Gateway。
- 交付物：权限引擎；沙箱运行时；安全门控清单；审核中间件。
- 验收：越权/注入用例被拦截；third_party 无法触达车控与非白名单网络；危险动作强制二次确认。

### WS9 · 可观测 & 评测 ‖ 规模 M ‖ 依赖：WS1
- 任务：OpenTelemetry trace 从 HMI→Edge→Cloud→Agent 端云贯通；指标（意图准确率、本地/云路由命中率、各 Agent 时延/成功率、LLM token/成本、降级触发率）；日志结构化 + 敏感脱敏；评测集（意图分类标注集、端到端场景回归集、Agent 契约测试）并入 CI 门禁。
- 交付物：可观测栈（OTel+Prometheus+Grafana+Loki/Tempo）；评测集 + CI 流水线。
- 验收：一次请求的端云全链路 trace 可视；CI 在 PR 上跑契约 + 场景回归并作为合并门禁。

### WS10 · 端侧工程化预备 ‖ 规模 M ‖ 依赖：WS1（部分过渡到 Phase 2）
- 任务：Fast Intent 规则 + 端侧轻量分类模型；意图阈值与本地白名单 OTA 下发机制（仅机制，不含生产发布）；端侧车控/媒体 Agent 从编排器拆分为独立模块。
- 交付物：端侧分类模型 v0.1；配置下发通道；端侧 Agent 模块化。
- 验收：端侧意图准确率达基线；阈值可热更新；端侧 Agent 可独立测试。

## 3. 里程碑与排期

| 里程碑 | 目标 | 包含 WS | 退出判据 |
|---|---|---|---|
| **M1 基础贯通**（~第1–5周）| 端云流式稳定、契约/可观测就位 | WS1、WS4、WS2(核心)、WS9(基础) | 端云流式贯通；CI 跑契约测试；一次请求全链路 trace 可见 |
| **M2 能力与体验**（~第5–11周）| 全能力域真实接入 + 多轮 + 语音 + 画像 | WS3、WS5、WS6、WS7 | 4 core + 6 eco 真实可用；组合意图与多轮可用；语音闭环 |
| **M3 安全与生态**（~第10–16周）| 权限/沙箱/合规 + 灰度 + 评测门禁 | WS8、WS2(灰度)、WS9(评测门禁)、WS10 | 权限强校验与沙箱生效；灰度发布可用；场景回归 ≥95% 并作为门禁 |

> WS 间存在并行：M2 的 WS6 各 Agent 可多人并行；WS5/WS7 与 WS3 并行推进。排期为相对周序，需按外部接口就绪度（R-A）滚动调整。

## 4. Phase 1 退出标准（DoD）

1. 4 个 core + 6 个 eco Agent 接真实/沙箱能力，黄金用例全通过。
2. 新增 Agent 仅通过注册接入，**0 改编排核心代码**（回归验证）。
3. 跨域组合意图、多轮澄清、二次确认端到端可用。
4. 端云双向流式稳定；§3.3 降级矩阵全部用例通过（断网车控不中断）。
5. 权限强校验 + third_party 沙箱生效；车控动作全部经 VAL 且可审计。
6. 语音 ASR/TTS 全链路闭环，可打断。
7. OTel 端云全链路追踪；核心指标看板上线。
8. 评测集（意图准确率 + 端到端场景回归）并入 CI，场景回归通过率 ≥ 95% 且作为合并门禁。
9. 画像合规闭环（可导出/删除），敏感数据上云最小化。

## 5. 风险与缓解

| # | 风险 | 影响 | 缓解 |
|---|---|---|---|
| R-A | 外部接口（地图/媒体/支付/ASR-TTS/车辆状态）就绪时间不可控 | 阻塞 WS6/WS7 | 适配层先行、mock 与真实双实现；按就绪度滚动排期；接口契约尽早冻结 |
| R-B | LLM 编排时延/成本超预期 | 体验/成本 | LLM Gateway 缓存 + 小模型分流 + 流式首响；规划与聚合分离可分别优化 |
| R-C | 意图准确率依赖真实数据 | 路由质量 | 建标注与线上回流闭环（WS9）；阈值可 OTA（WS10） |
| R-D | 端云通道弱网稳定性 | 可用性 | 断线重连 + 幂等 + 降级矩阵（WS4）；弱网/混沌注入测试 |
| R-E | third_party Agent 安全边界 | 安全 | 沙箱 + 权限白名单 + 审核（WS8）；商用审核流程留 Phase 2 |
| R-F | 车控 action 端云回流的安全语义 | 安全 | 云端只产意图，端侧 VAL 终校验（WS3+WS8）；车控真实对接留 Phase 2 |

## 6. 执行建议

- 每个 WS 建独立 spec→plan→实现 循环；本计划是总纲，单 WS 启动前用 brainstorming/writing-plans 细化。
- M1 是地基（契约/通道/可观测），务必先稳；M2 可大规模并行；M3 收口安全与质量门禁。
- 每条 `TODO(Phase1)` 注释对应本计划某个 WS，可用 `grep -rn "TODO(Phase1)"` 跟踪覆盖度。
