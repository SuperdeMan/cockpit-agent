## 1. 背景（为什么做）

roadmap 卡：「collector 增 `/metrics`（Prometheus 格式，聚合现有 NATS 指标）；可选 OTLP span
导出开关；compose 加 prometheus+grafana profile 与最小 dashboard json。验收：Grafana 能看到
Agent 时延/成功率/熔断状态曲线。」`docs/design/2026-06-15-observability-dashboard.md` 当初明确

预留扩展点的正式落地，不与原设计冲突。**验收标准只覆盖 Prometheus/Grafana 部分，OTLP 导出开关
没有单独验收**，给了这部分更大的实现自由度。

**核心边界**：不改 `orchestrator/cloud/{engine,dispatch,loop,circuit}.py`、
`observability/metrics.py`、`agents/_sdk/*`、`observability/collector/store.py` 任何一行
现有逻辑——`/metrics` 和 OTLP 桥接都只读已有数据，不改数据产生链路。

## 2. 关键发现（已核实，非设计阶段假设）

1. `store.agents[agent_id]` 已有全部所需数据：`{healthy, fail_count, last_seen, count,
   avg_ms, error_rate, deployment, kind, circuit}`（`circuit∈{closed,open,half_open}`，由
   `dispatch.py:76-81` 合并 `CircuitBreakerManager.snapshot()` 进 metric 上报）。`/metrics`
   只需遍历这个 dict，不需要碰 `circuit.py`/`dispatch.py`。
2. **`observability/tracing.py::setup_tracing()`（99行）是完整实现但从未被调用的死代码**——
   全仓 grep 只有定义处命中。它已经做好"有 `OTEL_EXPORTER_OTLP_ENDPOINT` 时建真 OTel
   Provider+OTLPSpanExporter，否则安全降级"。OTLP 桥接直接复用它。
3. span 事件确有 `ts` 字段（`observability/events.py:110` `_emit()` 对每个事件
   `setdefault("ts", _now_ms())`），约等于 span 上报（结束）时刻，供 OTLP 桥接反推近似
   start/end 时间。
4. **`parent_id` 在云端编排的真实调用点几乎从不被填**——全仓 `orchestrator/` 下 grep
   `parent_id` 零命中，唯一传值处是 `agents/_sdk/agent_client.py`/`http.py` 的 guardrail/
   provider 熔断窄路径。"扁平但同 trace 分组"的桥接策略如实反映现状，不是偷懒简化。
5. `grpcio==1.66.2`/`protobuf==7.35.0` 已在 7 个 requirements.txt 统一钉住，新增 opentelemetry
   依赖需与之兼容。
6. `server.py`/`store.py` 用包内相对导入（`from .store import CollectorStore`），新模块跟随；
   `main.py` 用绝对导入且**已有 `@app.on_event("startup") async def _startup()` 钩子**，
   只加一行调用即可，不重新定义整个函数。
7. 本仓**从未用过 Docker Compose `profiles:`**（全仓 grep 零命中）——本卡是第一次引入，但
   完全匹配卡面字面要求，是 Compose 标准机制，默认 `make up` 行为不变。

## 3. 方案

### 3.1 `/metrics` 端点——手写 Prometheus 文本格式，零新依赖

新建 `observability/collector/metrics_export.py`（纯函数，不 import FastAPI/NATS，独立可测）。
不引入 `prometheus_client`：数据源是"周期性整体覆盖式快照"（`store.agents` flat dict）而非需要
库维护状态的累加计数器；标签值是封闭词表（agent_id/deployment/kind），库主要防的转义风险不存在；
`count` 是外部系统算好的绝对值，用 `Counter` 反而要摸私有接口。

指标：`cockpit_agent_calls_total`(counter) / `cockpit_agent_latency_seconds_avg`(gauge，
avg_ms 转秒符合 Prometheus 基本单位命名规范，保留 `_avg` 后缀不假装是直方图) /
`cockpit_agent_error_rate`(gauge，不重复暴露 success_rate，PromQL `(1-error_rate)*100` 零成本
派生) / `cockpit_agent_circuit_state`(gauge，`closed=0<half_open=1<open=2` 单一有序数值配合
Grafana value mappings，而非 enum 多序列——验收只要求一条曲线) / `cockpit_agent_healthy`(gauge)
/ `cockpit_agent_health_fail_count`(gauge)。`route_hits`/`degrade`/`llm_tokens` 白名单里有但
无调用点真正填充，排除在外。`server.py` 新增 `GET /metrics` 路由（`PlainTextResponse`，
`media_type="text/plain; version=0.0.4; charset=utf-8"`），相对导入跟随现有风格。

### 3.2 OTLP span 导出桥接——完全限定在 collector 内部

新建 `observability/collector/otel_bridge.py`：`init_otel_bridge()` 复用 `tracing.py::
setup_tracing()`（相对导入 `from .. import tracing`），`OTEL_EXPORTER_OTLP_ENDPOINT` 未设置
时直接 return（连 `import opentelemetry` 都不执行）。`export_span(event)` best-effort 把
NATS span 事件转成真实 OTel span：trace_id/span_id 用 sha256 哈希成 OTel 要求的 128/64-bit
长度（同一字符串 trace_id 恒定映射，保证同 trace 分组）；start/end 时间用 `ts`(≈结束时刻) 减
`duration_ms` 反推近似值；**不引入自定义 IdGenerator 做字节级父子链接**——现状 `parent_id`
本来就几乎不存在（发现4），投入产出比低，扁平挂在 hash(trace_id) 对应的隐式根 SpanContext 下
即可，每 span 打 `cockpit.trace_id/span_id/parent_id/attrs.*` 属性保留原始字符串 ID 供交叉
核对。依赖：`opentelemetry-api/sdk/exporter-otlp-proto-grpc==1.43.0` + 显式钉
`grpcio==1.66.2`/`protobuf==7.35.0` 对齐全仓版本。接线只加两行：`main.py` 既有 `_startup()`
里调 `otel_bridge.init_otel_bridge()`；`server.py::ingest_loop()` 的 `obs.span` 分支里
`store.apply_span(event)` 后加 `otel_bridge.export_span(event)`。

### 3.3 compose 改动——`profiles: ["observability"]` 门控

新建 `deploy/prometheus/prometheus.yml`（scrape `observability-collector:8092`，15s 间隔）+
`deploy/grafana/provisioning/{datasources,dashboards}/*.yml`（固定 `uid: cockpit-prometheus`
供 dashboard JSON 引用，不依赖 UI 手工关联）+ `deploy/grafana/dashboards/cockpit-agents.json`
（3 panel：延迟 timeseries / 成功率 timeseries / 熔断状态 state-timeline + value mappings）。
`deploy/docker-compose.yaml` 新增 `prometheus`(`prom/prometheus:v3.13.0`) + `grafana`
(`grafana/grafana:12.3.8`) 两个服务，均 `profiles: ["observability"]`，沿用现有
`x-restart: &restart` anchor，各自 healthcheck（wget），`depends_on` 用默认
`service_started`（`observability-collector` 现状无健康检查，不为此卡额外改动）。**不挂
TSDB/Grafana 数据持久化卷**——容器重建后历史清空，与"不做 trace/metrics 持久化"非目标一致。
`.env.example` 新增 `GRAFANA_ADMIN_PASSWORD=admin`（PoC 默认凭证，同 Postgres 密码先例）。
