## 1. 背景：两个问题是同一个问题的两面

### 1.1 问题 A：嘈杂环境垃圾进 LLM（现状防线与缺口）

R4.3 把语音从 push-to-talk 升级为「唤醒 + VAD + 免唤醒续问 + barge-in」后，**进入云端的
语音不再都有显式用户动作背书**。现有拒识防线全部在 HMI 本地、全部是**字面/长度判据**
（`utteranceHeuristics.mjs` + `voiceLoop.mjs`）：

| 防线 | 判据 | 拦得住 |
|---|---|---|
| `isFiller` | 去标点后整句由语气字组成（≤6 字） | 嗯嗯 / 啊 / 哈哈 |
| 短语音噪声 | speech <300ms 且 ≤2 字 | 咳嗽、碰撞声的 ASR 残字 |
| `dismissMinChars` | 定稿 <2 字 | 单字噪声 |
| 退出/dismiss 词 | 占据整句+slack | 退下吧 / 没事了 |
| 误唤醒静默回收 | 唤醒后 5s 无 speech | 纯误唤醒 |
| 回声指纹 + 自触发降级 | 定稿与 TTS 文本重合 | 自己播报触发自己 |

**缺口**：≥3 字、成句的环境人声全部漏过——乘客间对话、电台/播客/短视频声音、副驾打电话。
两个最大入口是 **FOLLOWUP 续问窗**（TTS 播完后 8s 内任何 VAD speech 即进聆听，无唤醒门槛）
和 **barge-in**（SPEAKING 期 VAD ≥300ms 即打断转聆听）。这些句子上云后**云端零拒识**，去向三种：

1. **最糟**：被 Planner 规划成真任务——乘客说「我明天想去吃火锅」→ nearby/navigation 真执行；
   电台播「本台记者深入调研发现…」恰好命中 deep-research 的 route_hints pattern（深入+调研）
   → replace 成 research.run 跑分钟级深调研；
2. **次糟**：落 chitchat 全局兜底，一本正经回答乘客对话/电台新闻——车机突然插话，被打扰感强、
   隐私观感差；
3. **最轻**：`steps=[]` → 「抱歉我暂时无法处理」——也是一次不该发生的打断 + 一次 LLM 调用。

且垃圾轮会 `append_turn` 落对话历史、触发 memory 画像抽取——**污染指代消解与用户画像**。

### 1.2 问题 B：Agent 增多后路由错误（现状与缺口）

路由三层：端侧 `fast_intent`（规则，不中上云）→ 云端 Planner LLM（catalog 语义预筛 K=20 →
JSON DAG → `_validated_steps` 校验，重试 1 次）→ `RouteHintEngine`（manifest route_hints
确定性兜底）；全失败 → `_fallback`（chitchat 或 registry 语义 top-1）。

**缺口：路由不确定性没有任何显式表达与出口**——

- Planner LLM 拍板即执行。历史真机 bug 全是这个形态：「巴西那场帮我看看详情」被 nearby.detail
  抢（2026-07-06）、「珠海玩两天推荐景点」被 nearby.search 抢（同日）、「像笋的楼」被 Planner
  臆断成京基100（2026-07-07）。每次都靠**事后**加 guard/hint/校正器打补丁——补丁修的是已知
  case，**下一个长尾歧义句仍会先猜错一次**；
- `_fallback` 语义 top-1 自 R4.1 起有真实相似度分（`ResolvedAgent.score`），但**无门槛**：
  score 0.36 也硬执行该 Agent 的 `capabilities[0]` + 空槽位；
- 已有的 NEED_SLOT/NEED_CONFIRM 是 **Agent 级**机制（计划已定、缺槽位/要确认）；
  **路由级**「你是想 A 还是 B」不存在。

### 1.3 统一框架：置信度三段式

两个问题共享同一个本质：**系统对「这句话该不该处理、该给谁处理」没有置信度概念，只有
『硬着头皮处理』一条路**。本卡把响应策略显式分为三段：

| 置信度 | 响应 | 今天已有 | 本卡新增 |
|---|---|---|---|
| 高（意图明确） | 直接执行 | fast_intent / Planner / route_hints | —（一个字节不动） |
| 中（是请求，但落法歧义） | **澄清一句再执行** | 仅 Agent 级 NEED_SLOT | 路由级 `clarify` + `intent_choice` 卡 |
| 低（大概率不是对助手说的） | **拒识（静默丢弃）** | 仅 HMI 字面判据 | 语义级 `addressed` 判定 + 端侧自适应收紧 |

三段式的判定信号**不引入任何新模型**：受话判定与歧义申告合并进 Planner 既有的一次 LLM 调用；
fallback 门槛复用 registry 真语义分数（R4.1 资产）；端侧自适应复用 voiceLoop FSM 与
voiceMetrics（R4.3 资产）。

## 2. 目标 / 非目标

**目标**
- G1（拒识）：hands-free 开启 + 嘈杂环境（播客/乘客对话背景）下，非受话语句**静默不响应、
  不落对话历史、不进画像**；正常指令零回归（评测集量化，误拒率优先于漏拒率）。
- G2（澄清）：真歧义句（评测集定义）一次澄清后正确执行；**明确句零澄清**（39 条策展基线 +
  clarify 反例全绿——澄清滥用视为回归）。
- G3：连续拒识时端侧自适应收紧（续问窗减半 → 仅唤醒词），一次成功交互即恢复——嘈杂环境
  体验闭环不靠用户手动关开关。
- G4：全链路 fail-open + env 门控；push-to-talk/文本输入**永不被拒识**。

**非目标**
- 声学级受话检测（声纹、多音区拾音、视线检测）——车规量产件，PoC 浏览器不做（见 §9 量产映射）；
- 端侧 NLU/受话分类模型（R4.1b 的 K6 决策依旧 defer；本卡的 `addressed` 接口面为其预留）；
- 多意图句的**局部**歧义澄清（整句级歧义才澄清；「打开空调顺便找个川菜馆」里川菜馆歧义不问）；
- ASR 引擎置信度接入（三引擎协议未透传词级置信，P3 探针项）；
- L4 全双工（R4.3 §2 已钉死非目标）。

## 3. 关键决策

### D1 受话/歧义判定合并进 Planner 的同一次 LLM 调用（不加独立模型、不加独立调用）

`_PLANNER_SYSTEM` 输出 schema 扩展为：

```json
{"addressed": true|false,
 "complexity": "simple|adaptive", "goal": "...", "steps": [...]}
// 或（互斥于非空 steps）
{"addressed": true,
 "clarify": {"question": "您是想找附近的川菜馆，还是导航去某一家？",
             "options": [{"label": "找附近的", "send_text": "帮我找附近的川菜馆"},
                         {"label": "导航过去", "send_text": "导航去最近的川菜馆"}]}}
```

- **为什么合并**：延迟零增量（同一次调用多两个字段）、上下文同源（Planner 本来就看得到
  历史/焦点/画像召回）、成本零边际。独立小模型/独立 LLM 调用要么加延迟要么加成本，且
  上下文要重复装配。
- **弱 LLM（MiMo）判不稳怎么办**：①字段全可选——`addressed` 缺省 `true`、`clarify` 格式坏
  即丢弃，**fail-open 到今天的行为**；②D8 评测集先建尺、分 provider 出报告；③prompt 取保守
  取向——`addressed` 拿不准输出 `true`（宁可处理不可误丢）、`clarify` 明确句绝不允许反问。
- **prompt 关键界定**（执行时逐条落进 system）：
  - `addressed=false` 的典型：乘客间对话片段、自言自语、电台/视频/新闻播报腔、称呼他人
    姓名的话、与上下文完全无关且不构成请求的残句；
  - **情绪表达是受话**（「好烦啊」「唉」→ chitchat 共情是产品能力，不是垃圾）；
  - **缺槽位不算歧义**（「导航」缺目的地 → 交给 Agent NEED_SLOT 补槽，不出 clarify——
    不抢 F12 的活）；
  - 澄清仅当「同一句在能力清单上有 ≥2 种合理且结果差异大的落法、上下文无法裁决」；
    `send_text` 必须是消歧后**完整可执行的第一人称说法**；options ≤3、label ≤10 字（要 TTS 念）。
- 现有规则「宁可不答也不要张冠李戴」（`planning.py` 通用规则末条）升级为三级出口：
  确定 → steps；是请求但落法拿不准 → clarify；不是请求 → addressed=false。

### D2 拒识只作用于 hands-free 来源；fast_intent 命中天然豁免

- HMI 发送时带 `meta.input_source`：`voice_wake` / `voice_followup` / `voice_bargein`
  （VoiceLoop 已区分三条进入路径）+ `voice_utterance_ms`（本轮 speech 累计时长，FSM 已统计）。
  push-to-talk、文本输入、确认/补槽续接**不带**该 meta → 云端**永不拒识**（用户显式动作 =
  受话确定；挂起会话续接的短答「西湖」「第二个」天然豁免——它们走 pending session 分支，
  根本不进新规划路径）。
- 端侧 `fast_intent` 命中的指令在 edge 直接执行、不上云 → 嘈杂环境下车控/媒体**零影响、
  零延迟增加**。这同时是已知边界：环境音恰好被 ASR 转成完美指令（「打开空调」）时文本层
  无解——这是声学层（KWS 门槛/量产声纹）的职责，且危险车控本就有 `require_confirm` 兜底。
- meta 通路现成：HMI `dispatch(text, isConfirmation, location, metaExtra)` →
  `HandleRequest.meta` → edge 网关透传 → cloud（`nearby_poi_id` 先例）。

### D3 拒识反馈：`ui_card {type:"rejected"}` 免改 proto；不落对话历史、不进画像

- `FinalResult` 没有 meta 字段（speech/ui_card/actions/follow_up/need_confirm），**不加**——
  拒识标记放 `ui_card` Struct：`{"type":"rejected","reason":"not_addressed"}`，`speech` 置空
  （**不播 TTS**）。
- HMI 收到 `type=rejected`：移除本轮 pending 气泡（不渲染任何回复）、不播报、orb 微反馈
  （一次短暂脉冲，可选）、`voiceMetrics` 计数、通知 `handsFreeController`（D4 自适应输入）。
  用户气泡保留但标灰（「已忽略：疑似环境人声」悬浮提示）——**留痕迹供用户纠错**：如果确实
  是对助手说的，用户看得到它被忽略了，重说即可（聆听窗还在，纠错成本一句话）。
- **不落库**：`engine.handle()` 的 `append_turn` 在编排循环**之后**（`engine.py:75-80`），
  `_orchestrate` yield 的 rejected final 带内部标记 → `handle()` 跳过本轮 user+assistant
  落库 → 不进对话历史（不污染指代消解）、不触发 memory 画像抽取（不污染画像）。时序天然
  支持，零结构改动。

### D4 端云自适应闭环：连续拒识 → 收紧聆听 → 仅唤醒词 → 成功即恢复

嘈杂是**环境状态**不是单句事件，端侧要有会话级策略（全部在 `handsFreeController` +
`voiceLoop`，纯逻辑 node 可测）：

| 连续云端拒识 | 动作 | 恢复 |
|---|---|---|
| 1 次 | 仅计数 | 任一轮正常处理（非拒识 final）即全部复位 |
| ≥2 次 | `followupWindow` 临时减半（8s→4s）；notice「周围有点吵，我收紧了聆听」 | 同上 |
| ≥3 次 | 本会话降级**仅唤醒词模式**：FOLLOWUP 窗=0（播完直接回 ARMED）、barge-in 仅认 KWS 不认 VAD；notice「说『小舟小舟』再叫我」 | 同上；或用户开关 hands-free 重置 |

- 唤醒词是高精度信号（真麦验收命中可靠），嘈杂环境退化到「只认唤醒词」是正确姿势——
  比让用户手动关 hands-free 体验好一个量级。
- 本地字面判据（filler/短语音）**不计入**连续拒识（那些本来就没上云）；只有云端语义拒识
  才计数——避免安静环境下用户「嗯」两声就被收紧。

### D5 澄清的两个信号源：LLM 主动申告为主，fallback 分数门槛为辅

1. **主路径（LLM 活着）**：D1 的 `clarify` 输出——LLM 是唯一能看懂「这句话在当前能力清单上
   有两种合理落法」的组件，让它申告而不是让它硬猜。
2. **降级路径（LLM 挂/解析两次失败）**：`_fallback` 语义 top-1 现在无门槛硬执行
   `capabilities[0]`。P1 改为：`score < CLARIFY_FALLBACK_MIN`（默认复用
   `SEMANTIC_PROMOTE_SIM=0.5`）→ **不执行**，诚实降级「我没听清您想做什么，可以换个说法吗」。
   灰区（0.35~0.5）出确定性模板澄清卡的方案**推迟到 P3 可选**——LLM 挂掉时体验已降级，
   先把「低分硬执行首个能力」这个真 bug 堵上即可，不为降级路径过度设计。
3. **route_hints 不引入澄清**：正则命中=确定性高置信（它存在的意义就是压过弱 LLM 的犹豫），
   照旧 replace/append。

### D6 决策顺序（engine 消费面，一处收敛）

```
新规划轮（非确认/补槽续接）：
  1. REJECT 开 && meta.input_source ∈ voice_* && plan.addressed == false
       → 拒识短路（rejected final；route_hints 结果一并作废）
  2. plan.steps 非空（LLM 正常规划，或 route_hints 兜底命中）
       → 执行（今天的路径，含 D0 流式直通/T2 循环，一字不动）
  3. plan.clarify 非空 && CLARIFY 开 && 本轮非 clarify_resume
       → 澄清卡短路（speech=question + intent_choice 卡；不挂 session）
  4. 都空 → 今天的行为（"抱歉我暂时无法处理" / fallback 已在 build 内处理）
```

- **为什么拒识压过 route_hints**（1 在 2 前）：电台播报腔完全可能碰瓷领域正则（「深入调研」
  →research.run 跑分钟级流水线）。误杀真指令的代价=用户重说一遍（聆听窗还在）；不拒的代价=
  误执行+分钟级资源+被打扰。**代价不对称，拒识优先**。且该顺序只对 hands-free 源生效——
  显式输入不受影响。
- **为什么 steps 压过 clarify**（2 在 3 前）：route_hints 命中时 LLM 的犹豫作废——确定性
  优先的哲学与 R2.1 一以贯之。实现上 `build()` 内 `route_hints.apply` 照跑（对带 clarify 的
  空 steps 计划兜底加步），engine 只看最终 steps 是否非空，**planning 不需要感知 input_source**，
  改动面最小。

### D7 澄清载体：`intent_choice` 卡，零会话状态，深度=1

- 卡 payload：`{"type":"intent_choice","question":"...","options":[{"label":"...","send_text":"..."}]}`；
  `speech` 即 question（自动播报「…说第一个或第二个就行」）。
- **零会话状态**（对比 NEED_SLOT 挂 SessionState）：用户点选或说「第N个」→ HMI 把
  `option.send_text` 当**新请求**回发（附 `meta.clarify_resume=1`）。不选、直接说别的 →
  卡自然作废，正常处理。dest_choice/waypoint_choice 先例证明零状态够用：无 TTL、无话题
  切换判定、无孤儿清理，天然多轮安全。
- HMI 复用 `ordinalSelectIn` +「lastXXXRef 互斥消费」先例接住语音序号选择；卡片按钮点击
  直接 dispatch。
- **深度=1 硬规则**：`clarify_resume=1` 的轮次，planner 解析层直接丢弃 clarify 输出（若 LLM
  对消歧句仍犹豫 → 走正常校验/fallback）。防「问个不停」。
- 澄清不经 aggregator（engine 短路），无卡片优先级改动。

### D8 评测先行：先建尺，再动 prompt（照 R3.4 模式）

- `test/eval_corpus/rejection_cases.yaml`：
  - **正例（不得拒）≥25**：既有 8590 语料分域抽样 + 情绪/闲聊短句（「好无聊啊」）+
    嘈杂变形（句首尾粘 1-2 个噪声字模拟 ASR 污染）+ 上下文接续句；
  - **负例（应拒）≥15**：乘客对话片段（「妈你到哪了」「他昨天跟我说那个项目黄了」）、
    电台/新闻播报腔（含「深入调研」类 route_hints 碰瓷句）、TTS 自回声风格句、称呼他人
    姓名的话。
- `test/eval_corpus/clarify_cases.yaml`：

## 6. 兼容与回退矩阵

| 场景 | 行为 |
|---|---|
| hands-free 关（默认） | 无 `input_source` meta → 拒识链完全不激活；澄清（若开）仍生效 |
| `REJECT_NON_ADDRESSED=off` | planner 照常输出 addressed，engine 不消费——一键回今天 |
| `CLARIFY_ENABLED=off`（默认） | clarify 输出被解析层丢弃 → 行为=今天 |
| LLM 不输出新字段（弱模型/旧 prompt 缓存） | addressed 缺省 true、clarify 缺省无 → 今天的行为 |
| LLM 挂 / 解析两次失败 | `_fallback`：chitchat 优先不变；语义 top-1 加分数门槛（P1 后），门槛上行为=今天 |
| push-to-talk / 文本输入 | 永不拒识；澄清适用（歧义与输入方式无关） |
| 确认 / 补槽续接（F1/F12） | 不进新规划分支 → 拒识/澄清天然不适用，闭环零影响 |
| nightly / CI 纯 mock | mock LLM 不输出新字段 → fail-open 零感知；eval --live 自动 SKIP |

## 7. 风险

| 风险 | 缓解 |
|---|---|
| 弱 LLM addressed 误判，真指令被静默丢 | prompt 保守取向（拿不准=true）+ 正例评测 gate + user 气泡标灰可见可纠错 + 聆听窗内重说成本一句话 + env 可关 |
| 弱 LLM 滥用 clarify（明确句也问） | prompt 负例 + 39 策展「零澄清」硬断言 + 深度=1 + env 默认 off 灰度 + 分 provider 基线 |
| planner 输出 schema 变复杂 → JSON 坏率上升 | 字段全可选 fail-open + 解析失败率入 eval 报告对比改动前后 + 契约单测固化 |
| 拒识句实为补槽短答（「西湖」） | 补槽走 pending session 分支不进新规划 → 天然豁免；语料收录此类正例双保险 |
| 情绪/闲聊短句被误拒 | prompt 明确「情绪表达=受话」+ 正例语料收录 |
| 环境音被 ASR 转成完美指令 | 文本层无解，诚实为已知边界：fast_intent 端侧执行不受本卡影响，危险车控有 require_confirm；量产靠声学层（§9） |
| 澄清打断感（问太多） | 触发条件严格化 + 每轮最多 1 次 + 反例回归；上线后以 `clarify.shown/resolved/abandoned` 指标观测调阈值 |
| D6 拒识压过 route_hints 误伤 hint 域真指令 | 仅 hands-free 源生效；hint 域正例（真调研/行程指令）入 rejection 正例语料守护 |

## 9. 量产映射与已知边界

- `meta.input_source`（+utterance_ms）即**车机 DSP 音区/声纹置信的接口面**——量产替换为
  「主驾音区 + 声纹匹配分」时，云端消费面不变，只是信号从「进入路径枚举」升级为连续置信度。
- `addressed` 判定即**专用受话分类器（NLU 拒识模型）的接口面**——若 P3 触发 NLU 路径，
  engine 的 D6 消费逻辑一字不改，换的只是信号生产者（LLM 字段 → 端侧/边缘分类器）。
- 已知边界：①ASR 把噪声转成完美指令（声学层职责）；②多人同说导致 ASR 串音混句（多音区
  职责）；③浏览器 AEC 漏回声且文本不重合（现有回声指纹之外的残余）。三者 PoC 均不解，
  写入验收单观测项。
