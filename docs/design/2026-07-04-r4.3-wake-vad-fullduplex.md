## 1. 背景与现状

**现状语音链路**（截至 R4.1 落地后）：
- 输入：push-to-talk——按住光球（`Composer.tsx` + `audio.ts::StreamingRecognizer`）→
  `/api/asr/stream` WS 流式上屏 → **松手**定稿自动发送。三引擎可切（qwen3 realtime 带
  server_vad / fun-asr run-task / MiMo 分块回退）。
- 输出：句级批处理 TTS（R4.2 已设计流式化，未开工）；`stopTTS()` 仅在**发送新消息**时触发。
- 形象：`AuroraOrb` 三态 `idle | thinking | speaking`（`hmi/src/components/aurora/AuroraOrb.tsx`）。

**差距**：无唤醒词（必须触屏）；无客户端 VAD（结束靠松手，fun/mimo 引擎无任何端点检测）；
播报中用户开口无反应（不能打断）；一轮结束后想追问要重新按。WS7 的「唤醒与打断」自 Phase 1
计划立项以来一直空着。

## 2. 「全双工」分级定义与本卡范围

「全双工」一词歧义大，先拆级钉死范围，防止执行时理解漂移：

| 级 | 能力 | 本卡 |
|---|---|---|
| L0 | push-to-talk（现状） | 保留，永远是回退路径 |
| L1 | **免唤醒连续对话**：一轮回复结束后保持聆听窗 N 秒，用户接着说，VAD 断句自动发送 | ✅ P1 |
| L2 | **唤醒词 hands-free**：待机中说「小舟小舟」进入聆听，全程不碰屏 | ✅ P2 |
| L3 | **播报中打断（语音 barge-in）**：TTS 播放中用户开口 → 停播转聆听 | ✅ P3 |
| L4 | 真全双工（连续双向音频流对话模型，如 qwen-omni realtime，说听重叠/边听边答） | ❌ **非目标** |

L4 不做的理由要写清（这是架构决策不是工作量问题）：端到端语音对话模型直接从音频产生回应，
**绕开 Planner→确定性 Executor→VAL 的规划/执行分离链**（架构 §5.3 P0 安全要求），车控域不可
接受；且现架构是回合制编排。未来若做，只能作为 chitchat 单域实验另立卡，车控/任务域永远走
现有链路。

## 3. 目标 / 非目标

**目标**
- G1：hands-free 全链路可演示——「小舟小舟」→ 问天气 → 卡片+播报 → 播报中插话改问股票 →
  正确响应 → 沉默 8s 自动回待机；全程零触屏。
- G2：**唤醒前音频零上传**（KWS/VAD 全部浏览器本地 WASM 推理）；麦克风常开有显式视觉指示。
- G3：全部能力 **opt-in**（设置默认关）；关闭时行为与今天逐字节一致；nightly/CI 零感知。
- G4：语音回路状态机为**纯逻辑 `.mjs`、node 可测**（延续 pcmPlayer/ttsQueue/cardMath 的仓内惯例）。

**非目标**
- L4（见 §2）；自定义唤醒词训练（选型支持**运行时 pinyin 配置关键词**，不训模型）；
- 车规 DSP/NPU 唤醒引擎（浏览器 WASM 是 PoC 等价物；量产映射：本卡的 FSM/交互协议保留，
  KWS/VAD 引擎替换为车机 DSP 常驻件——分界面就是 §5 P1 的 worker 消息协议）；
- 多音区（主驾/副驾分离拾音）、声纹识别、离线 ASR（仍走云端流式 ASR）。

## 4. 技术选型与关键决策

### D1 唤醒 + VAD 引擎：sherpa-onnx WASM（一个运行时两个能力）

- **KWS**：sherpa-onnx 的 zipformer KWS 中文预训练模型（wenetspeech 系，Apache-2.0，模型
  ~3-6MB）。**关键词是运行时文本配置**（pinyin token 序列，如 `x iǎo zh ōu x iǎo zh ōu`
  即「小舟小舟」）——不需要训练，换唤醒词改配置即可。这是选它而非 Porcupine 的决定性理由
  （Porcupine 自定义中文词要在其 console 训练 + AccessKey + 商用许可不清爽）。
- **VAD**：同一 sherpa-onnx WASM 内置 silero-vad（业界事实标准，~2MB）。
- **备选链**（P0 探针失败时按序降级）：① Porcupine WASM 免费档（接受其 console 流程）；
  ② 砍 L2 只做 L1+L3（VAD-only，免唤醒会话改由一次点击开启）——范围收缩要回填本文件再执行。

### D2 运行位置：浏览器 Web Worker（隐私红线的技术落点）

`AudioWorklet` 采 16kHz mono PCM → `postMessage` 给 Worker（KWS+VAD 推理）→ 事件回主线程。
**唤醒前音频只存在于浏览器内存**；唤醒事件发生后才建 `/api/asr/stream` WS 上传语音。
每 utterance 一条 ASR WS（现状协议不动，唤醒→开、VAD 端点→关）；预热长连留作后续优化不入本卡。

### D3 端点检测（endpointing）双层分工

- **客户端 silero-vad 是权威端点**：speech-end（静音尾默认 800ms，可配）→ 定稿发送 + 关 ASR WS。
  这替代了「松手」，且让 fun-asr/MiMo 这两个**无 server VAD 的引擎第一次有了端点能力**。
- qwen3 的 server_vad 保持开启不冲突（引擎内断句归引擎，发送时机归客户端 VAD）。

### D4 语音回路状态机：`hmi/src/voiceLoop.mjs`（纯逻辑，核心交付物）

时间源/引擎事件全部注入（node 测试传 fake），UI 与 WASM 只是它的外设：

```
IDLE ──(hands-free 开)──▶ ARMED ──wake──▶ LISTENING ──VAD end + 定稿──▶ THINKING
  ▲                        ▲  ▲                                            │
  │(hands-free 关，任意态)  │  │                                        final 到达
  │                        │  └─FOLLOWUP 超时(默认 8s)                      ▼
  └────────────────────────┤                                           SPEAKING
                           │                                               │
                           └───────── FOLLOWUP ◀──TTS 播完──────────────────┘
                                        │
                                        └─VAD speech──▶ LISTENING（免唤醒追问）
SPEAKING ──barge-in（VAD speech ≥300ms 且过 D6 护栏）──▶ stopTTS + LISTENING
LISTENING ──唤醒后 5s 无 speech──▶ ARMED（静默回收，不发请求）
```

push-to-talk 与文本输入**不进 FSM**（原路径原样保留）；hands-free 关闭即整个 FSM 挂空。

### D5 误唤醒与「取消」的边界（两条必守规则，执行者必读）

1. **误唤醒静默回收**：唤醒后 5s 内 VAD 无 speech → 回 ARMED，不发任何请求、不出气泡。
2. **本地 dismiss 与云端 F1 的分界**：定稿文本 <2 字、或命中 dismiss 词表（没事/不是叫你/
   别听了/没叫你）→ **本地丢弃不上云**（防误唤醒把环境噪音句发给 Planner）。**例外**：HMI
   确认条可见（上一轮 `need_confirm=true`）时**一切定稿照发**——裸「取消/不要」必须到云端走
   F1 确认闭环。这承接 R4.1 §5.3 的红线：fast_intent 层做不到「知道有没有挂起确认」，而
   HMI 恰好持有 `need_confirm` 状态，**这是全链路里唯一能把这条规则做对的位置**。dismiss
   词表进 FSM 常量并配 node 测（含确认条可见时「取消」必须放行的用例）。

### D6 回声防御（L3 的成立前提）

- `getUserMedia` 显式 `echoCancellation: true, noiseSuppression: true`（浏览器 AEC 为主力）；
- SPEAKING 态判打断需 **VAD speech 持续 ≥300ms**（瞬时噪声不打断）；若此时 ASR partial 已
  返回，与正在播报的 TTS 文本做包含比对，命中即忽略（回声指纹，沿 R4.2 D6 思路）；
- KWS 在 SPEAKING 态保持有效（唤醒词几乎不会出现在 TTS 文本里；若某轮 TTS 文本含助手名
  「小舟」，该时段抑制 KWS 触发）；
- 连续 2 次疑似自触发（打断后定稿为空/与播报文本高度重合）→ 本会话自动关闭 L3 + toast 提示。

### D7 模型资产分发（吸取 R3.6 大文件下载的教训）

`hmi/public/models/` **gitignore**（+.gitkeep），新增 `scripts/fetch-voice-models.{ps1,sh}`：
主源（GitHub release）+ 国内镜像（hf-mirror.com）双源可切、`curl -C -` 断点续传、sha256 校验。
沿用 certs/ 的「gitignore + 生成脚本」先例。模型不可得 = P0 探针失败，走 D1 备选链，
**不允许**把模型二进制直接提交进仓（体积与许可卫生）。

## 6. 与 R4.2（流式 TTS + barge-in）的关系

- **取代**：R4.2 §3.4 的 v2「语音打断（ASR-partial 判据）」由本卡 P3 取代——VAD 判据比
  「等 ASR partial 返回」更早更稳，且不消耗 ASR 流量。R4.2 执行时 **v2 直接跳过**（其 §3.4
  已加注指向本卡）；v1 确定性打断（按光球即停）仍归 R4.2，且是本卡 L3 的逃生通道。
- **无硬依赖，有推荐顺序**：R4.2 P0-P2（流式 TTS）先做，打断后的「重新聆听→新回复」延迟
  更短、体验成型；但本卡对句级批处理 TTS 同样成立（P3 已写两种停播路径）。
- 共用风险面（回声/AEC）的护栏设计已在两卡间对齐（D6 ≈ R4.2 的回声指纹思路）。

- **技术链路通**：sherpa-onnx WASM VAD 引擎在浏览器本地跑通、驱动仓内真实 FSM（非模拟），
