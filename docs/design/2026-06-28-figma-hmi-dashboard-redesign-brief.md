## 0. 一页速读（TL;DR）

我们要给一套**云边协同的智能座舱 AI Agent 系统**重做两块界面的视觉：

1. **座舱 HMI**（车内主界面）——用户用**语音为主、触控为辅**完成车控、导航、信息、行程、深调研等任务的对话式界面。
2. **Dashboard**（开发/演示可观测台）——给工程/演示看"指令怎么走、车身有没有真变化、各 Agent 健康度"的大屏。

### 已锁定的四个方向（本次设计的前提）

| 维度 | 决策 | 含义 |
|---|---|---|
| HMI 目标画布 | **横屏中控大屏 1920×1080** | 左右分栏：左＝对话+卡片，右＝上下文可视化（地图/场景/车况）。不再是当前竖排网页列。 |
| 视觉方向 | **Aurora Glass · 极光液态座舱**（已定向，见 §8.0） | 提炼特斯拉/鸿蒙/小米/蔚来 NOMI/理想/奔驰 MBUX/宝马 iDrive + iOS 液态玻璃 + AI 极光炫光；旧"深空座舱 HUD"仅作现状参考基线。 |
| 范围 | **HMI + Dashboard 都重做** | 两者共用一套设计系统/tokens，但各有适配。 |
| 文档规定程度 | **需求 + 全量组件清单 + 约束 + 建议 IA 骨架** | 像素级视觉由设计师定；信息架构给到"分区骨架"作为起点。 |

### 设计师必须守住的"不可协商"（详见 §3、§9）

- **语音优先**：气泡承载"可朗读的结论"，卡片承载"证据"（来源/数据/时效/置信度），两者**不复读**。
- **行车安全（行车/泊车双态）**：行车态必须极简、单行、低眩光、可一眼扫读（NHTSA 单次注视 < 2s）；泊车态可展开详情。
- **状态完整**：每个组件都要给齐 **思考中 / 流式 / 过程区 / 待确认 / 主动播报 / 空 / 错误 / 离线 / 行车 / 日夜** 等态——这是 AI 座舱界面最容易漏设计的地方。
- **双主题 + 字号 + 大触控**：深色/浅色、标准/大字、标准/大触控均由 tokens 驱动，设计要成对产出。

> 一句话定位：这不是"车机版聊天机器人"，而是**一个会接地车辆与"我"、用语音给结论、用卡片给证据、在行车时克制、在泊车时展开的座舱副驾**。视觉要服务这个定位。

## 1. 产品是什么（设计师必读的语境）

智能座舱里的 AI 助手（默认昵称"小舟"）。用户一句自然语言，系统在**端侧"快系统"**（毫秒级处理车控/媒体等高频确定指令）与**云侧"慢系统"**（LLM 编排复杂、多轮、跨域任务）之间分级处理，结果流式返回到 HMI。

它能做的事（每一类都对应 HMI 上的不同呈现）：

| 能力域 | 例子 | HMI 呈现 |
|---|---|---|
| **车控**（端侧秒回） | "打开空调26度""主驾座椅加热""氛围灯调成绿色" | 气泡确认 + 动作卡；右侧可视化切到"车况场景" |
| **媒体** | "播放音乐""下一首" | 气泡 + 右侧"正在播放"场景 |
| **导航/出行** | "附近的充电站""导航去首都机场""导航去XX顺路吃饭" | POI 列表卡 / 路线卡 / 充电路线卡 + 右侧地图场景 |
| **信息** | 天气、预报、预警、空气质量、股票、新闻、赛事 | 天气卡 / 股票K线卡 / 新闻卡 / 赛事卡 |
| **联网搜索 & 深度调研** | "查一下…""帮我深入调研固态电池" | 证据卡 / 分节报告卡（可异步分钟级，完成后主动推送） |
| **行程规划** | "规划一个三天成都自驾" | 结构化多日行程卡（按天列停靠点+充电编织，可点导航） |
| **生态服务** | 点餐、停车缴费、用车手册问答 | 对应卡片 |
| **闲聊/情绪** | "讲个笑话""我今天有点不开心" | 纯气泡 |
| **主动服务** | 路况安全提醒、晨间早报、异步调研完成 | 主动气泡（💡）+ 可选报告卡 |

**关键产品哲学（直接影响视觉）：**

- **接地"车辆"与"我"**：助手知道当前位置、电量、行程、记住的偏好与常去地点。护城河不是"更强的聊天"，而是"结合此刻车辆状态与我的上下文给可落地的结果"。视觉应让"此刻的车/此刻的我"可感（如右侧场景区反映当前导航/媒体/车况）。
- **渐进式语音**：先给一句可朗读的结论，再给可展开的细节。行车时听结论，泊车时读详情。
- **诚实**：信息类结果带"时效"和"置信度"，资料不足会明确弃权/标注 gaps。视觉要给这些诚实信号留位置，而不是藏起来。

## 2. 用户与使用场景

### 2.1 谁在用
- **驾驶员**（行车中）：注意力在路面，**只能瞥**。语音为主，触控只点大目标。怕眩光、怕信息过载、怕需要二次确认时找不到按钮。
- **驾驶员/乘客**（泊车/充电中）：可以坐下来读长卡片（深调研报告、多日行程）、调设置、管理记忆与常用地点。
- **乘客**（行车中）：可深读，但主屏仍以驾驶员安全为先。

### 2.2 关键情境（每个都要在设计中显式覆盖）

| 情境 | 设计含义 |
|---|---|
| **行车态 vs 泊车态** | 行车：极简、单行、大字、低密度、禁止需要长读的展开。泊车：完整密度、可展开、可滚动长卡。**这是两套密度，不是一个缩放。** |
| **白天 vs 夜间** | 双主题（浅/深）。夜间低眩光、高对比但不刺眼；白天阳光下要保证可读对比度。 |
| **语音中 vs 触控中** | 语音：聆听动效、ASR 转写中、TTS 播报中（可见的"正在说"）。触控：大热区、即时反馈。 |
| **在线 vs 断网降级** | 断网仍可车控/媒体/基础问答（端侧兜底）。需要"连接中/已降级"的明确而不慌的视觉。 |
| **快响应 vs 慢任务** | 车控秒回；开放域/调研可能数十秒甚至异步分钟级。**慢任务必须有即时反馈→过程→结论的连续视觉**，杜绝"死寂等待"。 |

## 3. 设计原则与硬约束

### 3.1 七条设计原则（创作时的北极星）
1. **可一眼扫读（Glanceable）**：行车态信息分层清晰，关键信息 < 2 秒可读完。优先级：结论 > 关键数字 > 证据 > 元信息。
2. **语音优先、视觉补全**：界面是语音的"可视化回执"，不是必须读完的文本。气泡＝结论，卡片＝证据，**绝不复读**。
3. **接地此刻**：右侧可视化区随对话切换"场景"（地图/媒体/车况/待机），让"此刻的车"始终在场。
4. **克制的动效**：动效用于"状态转换可理解"（思考→流式→完成、车身值变化），不是装饰。尊重 `prefers-reduced-motion`。
5. **诚实的信息**：时效、置信度、资料 gaps、mock/降级标记都要有体面的位置，不藏不夸。
6. **大目标、宽容错**：触控目标 ≥ 当前 `--tap`（标准 48px / 大触控 60px）；危险动作必须二次确认且确认/取消都好按。
7. **成系统而非成页面**：所有视觉沉淀为 tokens 与组件变体，便于双主题/字号/触控派生，也便于后续代码 1:1 落地。

### 3.2 安全与架构红线（违反即不可用，来自 CLAUDE.md / 架构文档）
- **车控只经 VAL**：HMI 不直接操作车身；它只发意图、显示回执。Dashboard 的"对照实验"也走与 HMI 相同入口，**绝不新增绕过 VAL 的车控通路**。设计上不要出现"直接拨动车身开关"的错觉控件（除非明确是"建议指令"芯片）。
- **危险动作二次确认**（`require_confirm`/`need_confirm`）：必须有清晰的确认条（确认/取消），且行车态也要可安全操作。
- **敏感数据最小化**：精确位置、车内音视频、支付凭证默认不出车。设置里"定位/记忆/常用地点"的呈现要传达"可控、可删、可关"。
- **Dashboard 的 debug 只设环境量**（车速/电量/挡位/位置），**绝不写**空调/车窗等车控输出——滑块/控件的视觉语义要与"真实车控开关"区分开。

## 4. 目标画布与技术边界

### 4.1 画布
- **HMI 主画布：1920×1080（横屏 16:9）**。这是设计的"基准帧"。
  - 安全区/留白：四周预留 ≥ 24px 安全边距（车机不同屏会有切边/圆角）。
  - 同时产出**自适应行为说明**（不必做满所有断点，但要说明：内容列与可视化区在更窄屏如何收起/堆叠）。
- **Dashboard 主画布：1920×1080（横屏）**，已是横屏两栏，沿用横屏但重做视觉。

### 4.2 主题与可访问性变量（必须成套设计）
现有实现用 `<html>` 上的三个 data 属性驱动 CSS 变量切换，新设计请沿用这套"开关"心智，便于代码落地：

| 开关 | 取值 | 影响 |
|---|---|---|
| `data-theme` | `dark` / `light` | 整套色板切换（含语义色：涨跌、预警、置信度、空气质量分级等都要双版本） |
| `data-font` | `normal` / `large` | 全局字号基准（现状 large = 18.5px 基准）。大字下布局不能溢出/截断。 |
| `data-touch` | `normal` / `large` | 触控热区与栏高（现状 `--tap` 48→60、`--bar-h` 64→72）。 |

> 设计交付要给出：dark×normal 为基准帧；至少再给 **light 主题**完整一套、**大字/大触控**关键屏的适配示例。

### 4.3 技术栈边界（影响可实现性，请勿设计无法落地的元素）
- HMI / Dashboard 均为 **React + TypeScript + Vite**，纯 Web 渲染（车机内置浏览器）。
- 字体经 Web 字体加载，**离线/车机内自动降级到系统字体**——所以**不要把信息编码在某个特定字体的独有字形上**；中文需可用 PingFang/HarmonyOS/Microsoft YaHei 兜底。
- 图标现状是 **Emoji**（🚘🎵🧭❄️…）。可以提议替换为一套**自绘线性图标**（更可控、更高级），但需在交付物里给全图标清单（见 §6 各处出现的 icon）。若保留 emoji 风格也可，但要统一。
- 卡片里的**股票 K 线是 SVG 自绘**、赛事队徽是**远程位图**、可视化地图建议用**地图 SDK 占位**——设计要标注哪些是"数据驱动绘制区"。
- 性能：玻璃态 `backdrop-filter`、大面积模糊、持续动画在车机 SoC 上可能掉帧。**可以用，但要克制**，并为低性能场景给"降级版"（纯色面板替代实时模糊）。

## 5. HMI 信息架构（建议骨架）

> 这是"起点骨架"，不是最终布局。设计师可重组，但需覆盖同样的功能与状态。

### 5.1 顶层分区（1920×1080 横屏）

```
┌──────────────────────────────────────────────────────────────────────┐
│  ① 状态栏 StatusBar  —— 品牌/助手名 · 连接&模型态 · TTS开关 · 设置入口   │
├───────────────────────────────────┬──────────────────────────────────┤
│                                   │                                  │
│  ② 对话流 Conversation             │  ③ 上下文舞台 Contextual Stage     │
│     - 用户/助手气泡                 │     随对话切换"场景"：             │
│     - 思考中 / 流式 / 过程区        │     · 地图场景（导航/POI/路线/行程）│
│     - 信息&动作卡片（约20种）        │     · 媒体场景（正在播放）          │
│     - 确认条 / 跟进提示 / 主动播报   │     · 车况场景（空调/座椅/氛围灯）   │
│     （可纵向滚动）                  │     · 待机/氛围场景（无任务时）      │
│                                   │     · 大卡聚焦（长报告/行程占满舞台）│
├───────────────────────────────────┴──────────────────────────────────┤
│  ④ 输入区 Composer  —— 快捷指令轨 · 麦克风(按住/点按) · 文本输入 · 发送   │
└──────────────────────────────────────────────────────────────────────┘
```

**比例建议（可调）**：左对话列约 40–46%、右舞台约 54–60%。对话列保证气泡+卡片可读宽度；舞台保证地图/报告有舒展空间。

### 5.2 上下文舞台（③）的"场景"机制 —— 这是横屏带来的最大设计机会
当前竖屏把一切塞进一条聊天流。横屏后，右侧应是一个**随对话内容切换的"舞台"**，让"此刻的车"可见：

| 场景 | 触发 | 内容要点 |
|---|---|---|
| **地图场景** | 导航/POI/充电/路线/行程类意图 | 地图 + 路线 + POI 标点 + 途经/充电点；可与左侧 POI 列表卡联动高亮 |
| **媒体场景** | 媒体播放 | 封面/曲目/进度/音量；大而美的"正在播放" |
| **车况场景** | 车控（空调/座椅/氛围灯/车窗…） | 当前被调整的车身部件可视化 + 实时值（呼应用户刚下的指令） |
| **待机/氛围场景** | 无活跃任务 | 时间/天气/车辆概览/氛围动画；体现品牌气质 |
| **大卡聚焦** | 深调研报告/多日行程等长内容 | 让长卡占满舞台从容阅读（泊车态） |

> 设计要给出：场景切换的转场逻辑与默认（待机）场景。若设计师认为某些卡片更适合放右舞台（如行程、报告），请在稿中明确"哪些卡片在左列、哪些升舱到右舞台"。

### 5.3 行车态布局变体
行车态（系统按 VAL 标注 `driving=true`）下，整屏切到**低密度变体**：右舞台默认地图/极简，左列只显结论与极简卡，过程区强制单行不可展开，字号与热区放大。请把行车态当作**一等布局**来设计，而非泊车态的临时遮罩。

## 6. HMI 组件与屏幕全量清单（设计范围 checklist）

> 数据字段来自 `hmi/src/types.ts`（单一事实源）。设计每个组件时，请覆盖列出的**字段**与**状态**，并对"可缺省字段"设计优雅降级（字段缺失时不留空洞）。

### 6.1 全局框架

| 组件 | 内容 / 字段 | 状态 |
|---|---|---|
| **StatusBar** | 品牌标记、助手昵称（`assistantName`，用户可改）、连接态（在线/连接中）、模型态（快速/深度推理/自动）、TTS 开关（🔊/🔇）、设置入口（⚙） | 在线 / 连接中(断线重连) / TTS 开 / TTS 关 |
| **Composer 输入区** | 快捷指令轨（`quickCommands`，最多12条横向滚动芯片）、麦克风按钮、文本输入框、发送按钮；可显示提示（如"正在连接座舱服务…"/录音不支持提示） | 麦克风：空闲🎤 / 录音中🔴(声波动效) / 识别中◌；输入：空/有字/聆听中占位；连接提示条 |
| **欢迎态 Welcome** | 助手光球、"我是{昵称}"、副标题、3 个引导芯片 | 首次/空对话时 |

麦克风两种交互模式（设置可切）：**按住说话**（press-hold）/ **点按切换**（toggle）。两种都要有清晰的"正在录音"反馈。

### 6.2 对话流原子组件

| 组件 | 说明 | 必须设计的状态 |
|---|---|---|
| **用户气泡** | 右对齐 | 普通 |
| **助手气泡** | 左对齐 + 助手头像 | 普通 / 错误（`error`）|
| **思考中 ThinkingDots** | 慢响应即时反馈（"正在思考…" + 律动） | 单一动效态 |
| **流式文本 + 光标** | `speech_delta` 逐字填充 + 闪烁光标 | 流式中 / 流式结束 |
| **过程区 ProcessPanel** | 复杂任务的"步骤+思考摘要"折叠条，见 6.2.1 | 进行中 / 完成可展开 / **行车态单行不可展开** |
| **动作卡 ActionChip** | 车控等动作回执：`type` 标签 + 摘要 + "需确认"标记 | 普通 / 需确认 |
| **确认条 ConfirmBar** | 危险动作二次确认：确认/取消两个大按钮 | 待确认（仅最后一条且 `awaitConfirm`）|
| **跟进提示 followUp** | 一句话引导下一步 | 有/无 |
| **主动播报气泡** | `proactive`：💡 前缀的独立通知气泡（路况/早报/异步调研完成），可附报告卡 | 纯文字 / 带卡片 |
| **信息&动作卡片** | 见 6.3 卡片系统（约 20 种） | 各卡独立 |

#### 6.2.1 过程区（ProcessPanel）四阶段 —— 重点设计
复杂任务（联网搜索/调研/多 Agent 编排）会下发"过程"，分四阶段，**内容已后端脱敏**（只给步骤标签+摘要，不露原始思维链）：

1. **理解需求**（understand）2. **规划步骤**（plan）3. **执行任务**（execute，含多个子步骤，running→done 合并）4. **整理结果**（synthesize）

- **进行中**：折叠条标题"正在处理复杂任务" + 已完成阶段的 ✓ 摘要 + 进行中步骤"正在{label}…"。
- **完成后**：默认折叠为"处理过程（N 个步骤）"，可展开成四阶段编号时间线（执行任务下挂各子能力项）。
- **行车态**：强制单行、不可展开（安全）。

> 这是体现"AI 在认真干活"的关键，也是慢任务不"死寂"的核心。请把它设计得**可信、克制、可一眼判断进度**，并与"思考中""流式"形成连贯的三段叙事：思考中 → 过程区推进 → 流式/最终结论。

## 8. 设计系统 / Design Tokens（必须产出的"地基"）

> 视觉方向已定（见 §8.0 Aurora Glass），但**结构仍要成系统**。请把视觉决策沉淀为下列 token 族（命名可自定，但分类要齐全），这样后续 Claude Code 能 1:1 映射成 CSS 变量与组件，且双主题/字号/触控可派生。

### 8.0 视觉方向：Aurora Glass · 极光液态座舱（已定向）

> 经与泓舟对齐，本次不再"自由发挥"，而是确立一个明确方向：**Aurora Glass · 极光液态座舱**。下列 8.1–8.5 的 tokens 即服务这个方向。

**气质**：深邃 · premium · 有生命感 · 可信赖。提炼自主流高端座舱并叠加 iOS 液态玻璃与 AI 炫光——特斯拉（地图为底、极简克制）、鸿蒙/小米（卡片化、流畅深度与模糊）、蔚来 NOMI/理想同学（人格化 AI 化身·光球）、奔驰 MBUX（玻璃材质的奢华光泽、AI 浮现内容）、宝马 iDrive（仪表级精密、Shy-Tech）、iOS Liquid Glass（半透折射、镜面高光、连续圆角、自适应明暗）、AI 炫光（Apple Intelligence/Siri 边缘极光、Gemini 渐变）→ 我们的"极光签名渐变"。**严禁**：SaaS 后台感、廉价玻璃、彩虹乱用、过曝炫光。

**三条支柱**：
1. **活的场景做底**：最底层是地图/极光氛围（深空冷色基底 `#06080F → #0A0E1A` 渐变），玻璃面板悬浮折射其上——玻璃必须有"背后透出的内容"才成立。
2. **液态玻璃材质**：半透 + 背景模糊 28–40px + 轻微饱和提升 + 极淡冷色 tint(8–14%) + 镜面边缘（顶/左缘 1px 高光、底缘微暗）+ 悬浮投影/内顶高光 + 连续圆角(squircle)20–28px + 极淡边缘折射色散；浅色主题转"磨砂白玻璃"。给低性能降级版（纯色面板替代实时模糊）。
3. **AI 极光炫光（克制）**：签名光谱渐变 `#5BE9FF 青 → #5B8CFF 蓝 → #9A6BFF 紫 → #FF6BD6 品红` = "AI 专属光"。**只在 AI 有生命/AI 出品处**用：助手光球、聆听/思考的屏幕边缘极光辉光、流式文字虹彩光标与微光扫过、AI 生成内容（调研报告等）的虹彩细描边/角标、主操作虹彩描边。**铁律：正文与关键数字绝不上虹彩**（破坏对比）；其余交互用单一冷蓝实色 `#46D6E0`，把极光留给"AI 时刻"。

**品牌之魂——AI 化身"小舟"**：一颗液态玻璃光球、内部极光流动；待机慢呼吸、思考旋动、说话脉动、可朝内容微倾。贯穿欢迎/思考/主动播报，是这套设计的记忆点。

**可读性护栏（车规，硬约束）**：文字不直接压地图（下垫玻璃/scrim，≥4.5:1）；明亮地图区玻璃自动加深保对比；行车态降低模糊与辉光开销、提高不透明度与对比、让极光更安静；阳光下浅色主题 = 磨砂白 + 更强对比 + 收敛辉光。

### 8.1 颜色（每个语义槽位都要 dark + light 两版）
- **背景层级**：bg / bg-2 / surface（面板）/ surface-2（卡片）/ 实底（弹层）。

  - 涨跌：**红涨 / 绿跌（A股，勿反）**，dark+light 各一。

- 行高、字重、字间距（HUD 标题常用宽字距）。

### 8.3 间距 / 圆角 / 尺寸 / 层级
- 间距尺度（4 或 8 基准）、圆角尺度（现状 12/18/26）、触控尺寸 `--tap`（48/60）、栏高 `--bar-h`（64/72）、内容最大宽、卡片内距。
- 阴影/高度（elevation）层级：面板 / 卡片 / 弹层 / 高亮。

### 8.4 动效
- 时长与缓动曲线 token（入场、状态切换、强调闪动、长任务进度、声波/思考律动）。
- **"刚变"闪动**、**思考律动**、**流式光标**、**链路节点入场**、**录音声波** 这几个标志性动效请给明确规格。
- 全部受 `prefers-reduced-motion` 约束的降级说明。

### 8.5 图标
- 一套统一图标（建议自绘线性）。至少覆盖：天气族（晴/多云/阴/雨/雪/雾/霾…）、能力 Agent（10个）、车身部件（空调/座椅/车窗/天窗/氛围灯/大灯/雨刷/后备箱/充电口…）、信息族（搜索/新闻/调研/赛事/股票）、出行（POI/充电⚡/餐厅/酒店/景点）、系统（设置/麦克风/播报/连接/确认/关闭…）。给出清单与风格规范。

## 9. 无障碍与车规可读性（验收会查）
- **对比度**：正文文本对背景 ≥ 4.5:1，大字/图形 ≥ 3:1；浅色主题在阳光下、深色主题在夜间都要达标。三级灰阶不要让 `faint` 低于可读线（信息别"隐身"）。
- **触控目标**：≥ `--tap`（标准 48 / 大触控 60）；行车态关键目标更大、间距更宽，防误触。
- **一眼扫读**：行车态关键信息 ≤ 2 秒读完；信息分层（结论>数字>证据>元信息）在视觉权重上要体现。
- **不以颜色为唯一信息载体**：涨跌/置信度/状态除颜色外要有文字或形状（如 ▲▼、"高/中/低"、ON/OFF），照顾色觉障碍与强光。
- **动效可关**：尊重 `prefers-reduced-motion`，所有循环动画有静态降级。
- **文本可伸缩**：大字模式下不溢出、不截断关键信息。

## 11. 实现映射（写给后续按稿落地的 Claude Code）

> 设计师可略过本节；它确保设计能干净地变成现有代码结构。

- **消息驱动 UI**：WS 事件 `speech_delta | process | action | final | proactive | error`（见 `hmi/src/App.tsx` `handleEvent`）→ 对应气泡/过程区/动作/卡片/主动/错误。`Msg` 与各 `*Card` 类型是契约（`hmi/src/types.ts`），**视觉重做不应改这些数据字段**（改字段属于后端/契约改动，不在本次范围）。
- **右舞台（新）**：横屏新增的"上下文舞台"是**新增前端区域**，需要新组件 + 一个"当前场景"状态机（依据最近意图/卡片类型切换）。地图建议接地图 SDK；media/车况/待机场景由现有车辆状态与播放态驱动（部分数据已在 Dashboard 的车辆状态体系里，HMI 侧需补取数）。这是落地时**唯一较大的新增工作量**，设计稿请对它的数据来源与默认态给足说明。
- **Dashboard**：组件结构 `dashboard/src/components/{CommandBar,TracePanel,VehicleState,Dynamics,AgentList}.tsx`；车辆状态的分组/标签/渲染类型集中在 `vehicle-config.ts`（配置驱动，新增车身键改配置不改渲染）。
