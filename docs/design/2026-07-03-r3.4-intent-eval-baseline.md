## 1. 背景（为什么做）

全仓审计标出 G3：端侧 `fast_intent` 和云侧 `route_hints` 两套确定性路由规则，历史上已经因为
正则/关键词互扰出过真实回归（"深入调研固态电池"被裸「电池」劫持成电量查询、"第二天行程换一个"
里的"行"字被误判成确认词），但修复后只落了零散 pytest 断言，从未有一个聚合的准确率/召回率报告
或可比对的基线。全仓库 grep 确认没有任何地方计算过这类指标（`observability/metrics.py` 只在
文档字符串里提过"意图准确率/路由命中率"，从未实现）。

目标：为这两套路由机制建立可复现、可比对基线的评测基础设施，让"改坏一条正则"能在 CI 里被
观测到（先不阻塞合并，对应 roadmap 卡"③ CI 比对基线，跌破阈值告警（不阻塞，先观测）"）。

**核心边界**：新增一个只读评测器，不修改 `fast_intent.py`、`route_hints.py`，也不碰编排核心
四件套（`planning.py`/`context.py`/`aggregator.py`/`progress.py`）——只做三类新增：新数据文件、
新脚本、新 CI 步骤。

## 2. 关键发现：roadmap 卡面的数据源假设与仓库现状不符

调研（3 个并行 Explore + 1 个 Plan 设计 + 亲自核对关键文件）确认两处需要对卡面原文纠偏：

1. **"飞书 1465 意图库"标注语料实际不可得**：原始表 `feishu_tblN5NfQff850L5O_*.json` 已
   gitignore（`.gitignore:39`）且磁盘上不存在——只在生成 `commands.yaml`/`entities.yaml` 时
   用过一次（`scripts/gen_commands_yaml.py`），未保留标注语料（`commands.yaml` 1082 行里没有
   任何自然语句样例字段）。**决定：不重新拉飞书表**，v1 用现有可得数据源，报告里如实标注覆盖
   规模，把"补全飞书全量语料"列为后续增强项，不作为本卡验收阻塞。
2. **"88 条 corpus"里真正测「自然语言→意图分类准确率」的只有约 29 条**：`vehicle_objects.yaml`
   的 `val_execution`（36 条）测结构化指令→VAL 执行，`safety_gate.yaml`（23 条）测安全门控，
   两者都不经过 `classify_structured` 走 NL 解析——是另一个维度。真正相关的是
   `vehicle_objects.yaml::intent_recognition`（15 条）+ `multi_intent.yaml::split/no_split`
   （14 条）。已知的历史回归案例（电池劫持等）目前只以 pytest 内联断言存在
   （`test_fast_intent_extended.py::TestBatteryQuery`），未进入任何 corpus YAML 体系。

## 3. 方案：数据层怎么建、脚本怎么写

### 3.1 数据层——分层复用，不复制、不重复造轮子

只读引用既有语料（零拷贝，单一真相源）：
- `orchestrator/edge/tests/corpus/vehicle_objects.yaml::intent_recognition`（15 条）
- `orchestrator/edge/tests/corpus/multi_intent.yaml::split`+`no_split`（14 条）

新增 `test/eval_corpus/edge_regressions.yaml`：转录 `test_fast_intent_extended.py::TestBatteryQuery`
（396-430 行，4 个方法约 10 条用例）为结构化数据（`text`/`expect_object`或`forbid_object`/
`allow_none`/`source`/`tags`），**不删除原 pytest**（继续独立跑，第一道防线；本文件是把同一批
案例转成可复用、可聚合基线的第二道防线）。v1 范围只转录 battery 这组（roadmap G3 背景明确点名
的案例），其他 guard（`climate_feeling_intents` 等）留后续用同一 schema 扩展。

新增 `test/eval_corpus/route_hints_cases.yaml`：用「路由前后 plan 的 intent 列表」统一断言
（`initial_intents`/`expect_final_intents`，而非 `expect_intent`/`expect_no_route` 两套字段）——
因为 `RouteHintEngine.apply()` 的 `hit` 返回值有反直觉之处（"LLM 已路由对"和"guard 拦下"都会
导致 `plan.steps` 不变，但前者 `hit=True`、后者 `hit=False`），直接比较最终 intent 列表能用
同一断言口径覆盖 replace/append/guard 拦截/已正确路由/优先级冲突五种场景。案例来源：
`test_route_hints.py`（11 个引擎单测）+ `test_planning.py` 的 guard-rail 回归（`test_plain_
search_not_hijacked_by_research_net:211`/`test_does_not_inject_trip_for_plain_navigation:327`/
`test_does_not_inject_trip_when_planner_unavailable:472`）。

**明确排除** `_confirm_reply`（"行程含'行'"回归，`orchestrator/cloud/engine.py:394-409`）——
它是独立的整句长度启发式，不走 `manifest.route_hints` 声明式规则，`RouteHintEngine.apply()`
根本管不到；若要覆盖应是独立卡片。

**已核实的关键实现细节**：`RouteHintEngine(PlanBuilder._validated_steps)` 这个构造方式就是
生产代码本身——`orchestrator/cloud/planning.py:134` 的 `PlanBuilder.__init__` 写的正是
`self._route_hints = RouteHintEngine(self._validated_steps)`，且 `_validated_steps` 是
`@staticmethod`（`planning.py:259-260`）。评测脚本复用这条真实装配路径（含 intent∈capabilities
校验），比 `test_route_hints.py:10-15` 现有的玩具 `_validate` fixture（无校验拼 `Step(**raw)`）
覆盖更严格。

### 3.2 脚本/报告层

```
test/
  eval_common.py                # 报告构建/Markdown渲染/基线diff/CI告警公共工具（纯函数）
  eval_fast_intent.py             # edge 侧评测入口（卡面指定文件名）
  eval_route_hints.py             # cloud 侧评测入口
  eval_corpus/
    edge_regressions.yaml
    route_hints_cases.yaml
  test_eval_common.py             # eval_common 自身单测（合成数据，被 ci.yml 现有分组自动收集）
docs/reviews/eval/
  baseline_fast_intent.{json,md}  # --write-baseline 产出，人工审阅后提交
  baseline_route_hints.{json,md}
  README.md
```

指标不做成"一个数字打天下"：object_recognition（多分类：覆盖/召回+命中后准确率+整体正确率）
与 battery 负例（二元 guard-rail 通过率）分桶报告、不加权平均；cloud 侧同理分
`route_recall`/`route_guardrail` 两桶。

### 3.3 CI 接线

新增独立 job `intent-eval-baseline`（单一 Python 3.11，不进现有 matrix；不进 nightly——两套
逻辑都是纯规则引擎零 LLM/docker 依赖，秒级）。"跌破阈值告警"落地为**逐例回归比对**
（`regression_count > 0` 触发 `::warning::`），不是聚合百分比模糊容差——两套逻辑都是确定性
规则引擎，同代码同输入 100% 可复现，一个用例从基线 pass 翻 fail 就是唯一告警信号。默认
`main()` 不带 `--strict` 时无论如何 exit 0 + CI 步骤 `continue-on-error: true` 双保险，
`--strict` 预留接口 v1 不启用，对应 roadmap"不阻塞，先观测"。
