## 0. 实施者必读：本次调研核实过的事实与坑（动手前先记住）

1. **`_parse_and_validate` 会吞掉空 steps 计划**（`orchestrator/cloud/planning.py:221-257`）：
   现逻辑 `steps` 校验失败/为空 → `return None` → `build()` 重试 → `_fallback`。而
   `addressed=false` 与 `clarify` 的合法输出 **steps 恰恰为空**——若不改，这两个信号会被当
   「解析失败」丢弃并触发重试+fallback。**必须在 steps 校验之前先短路解析这两个字段**（见 T1）。
2. **`build()` 的重试条件同坑**（`planning.py:153-158`）：`if parsed and parsed.steps: break`
   ——要放行 `addressed=False` / `clarify` 非空的无 steps 计划（见 T1）。
3. **meta→`PlanContext.prefs` 是白名单制**（`orchestrator/cloud/context.py:426-430`）：
   新 meta 键 `input_source` / `voice_utterance_ms` / `clarify_resume` 必须显式加进白名单元组，
   否则 engine/planning 从 `ctx.prefs` 读不到。engine 统一从 `ctx.prefs` 读（不直接摸
   `request.meta`，与既有 prefs 消费一致）。
4. **engine 拒识短路必须在 `if not plan.steps:` 之前**（`orchestrator/cloud/engine.py:159-161`）：
   `addressed=false` 时 LLM 输出 `steps:[]`，若先走既有空计划分支会回「抱歉我暂时无法处理」
   并播 TTS——拒识失效。插入点=`planner.build` 返回之后、`if not plan.steps` 之前。
5. **rejected 标记经内部键 `_rejected` 传递**：engine yield 的 final dict 由
   `orchestrator/cloud/server.py:42-63` 组装 FinalResult——只取已知字段，`ui_card` dict 会被
   `final.ui_card.update(...)` 自动转 Struct（**卡片通路零改动可用**）。`handle()`
   （`engine.py:69-80`）在 yield 前 `ev.pop("_rejected", False)` 捕获，`rejected` 轮跳过
   `append_turn`（user+assistant 都不落——落库本就在编排循环之后，时序天然支持）。
6. **HMI rejected 分支必须调 `handsFreeRef.current?.turnEnded()`**：否则 FSM 卡 THINKING
   死锁（R4.3b 四死锁教训——App 每个终局分支都要放 FSM 出来）。rejected final 的 `speech`
   为空，早 return 会跳过既有 317-378 分支的 turnEnded 路径，必须自己补。
7. **route_hints 对空 steps 计划照样 apply 是 by design**（`planning.py:169`）：clarify 计划被
   hint 命中 → steps 非空 → engine 顺序自然「执行压过澄清」（母卡 D6-2>D6-3）；
   `addressed=false` 计划即使被 hint 填了 steps，engine 拒识分支也无视 steps 短路
   （母卡 D6-1 最优先）。**不要**为这两种情况在 planning 里加特判。
8. **Docker 无卷挂载**：改 `orchestrator/cloud/*` 后真栈验证必须
   `docker compose -f compose.yaml up -d --build cloud-planner`；改 HMI 必须 `--build hmi`
   （历史三次踩坑）。改 `orchestrator/edge/*`（本卡不涉及）同理 edge-orchestrator。
9. **nightly 纯 mock 的 LLM 不输出新字段** → fail-open 路径（addressed 缺省 true / clarify
   缺省无）必须有单测钉死；`e2e_rejection.py` 依赖真 LLM 判定，无凭据时 SKIP（沿
   `e2e_memory.py` 的 SKIP guard 先例）。
10. **prompt 按 env 拼接要做成函数**：`_PLANNER_SYSTEM` 改为 `_planner_system()`（每次
    `build()` 调用时拼，`os.getenv` 实时读）——`CLARIFY_ENABLED=off` 时 prompt **不含**澄清段
    （否则 LLM 输出 clarify 被 engine 丢弃 → 空计划话术，比今天更差）；受话段恒含（消费端
    按来源门控，无副作用）。函数化同时让 monkeypatch 单测可行。
11. **hands-free 定稿命中 HMI 早退分支（「第N个」候选选择等）不带 `input_source` 是正确行为**：
    `App.tsx::send` 的候选消费分支自行 `dispatch`（无 metaExtra）——选择句天然豁免拒识，
    不要「顺手」把 meta 加到那些分支。
12. 当前基线（2026-07-07）：全量 `python -m pytest --import-mode=importlib` **1146 passed,
    7 skipped**；HMI `node --test` **119**；改完不得低于此数（只增不减）。
