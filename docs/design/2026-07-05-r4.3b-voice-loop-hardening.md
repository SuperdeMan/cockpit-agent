## 2. 根因分析（逐条，含证据）

### 2.1 U1 首次唤醒双份识别/双份 TTS —— 孤儿活控制器

**触发前提**：`settings.handsFree` 已持久化为 true（上一会话开过），刷新/进入页面。

**链路**：
1. `hmi/src/main.tsx:40` 全 app 包在 `<React.StrictMode>`；hmi 容器跑的是 vite dev
   server（无卷挂载但 CMD 是 dev），StrictMode 在 dev 下对每个 effect 执行
   mount→cleanup→remount 双跑。
2. `App.tsx:130-167` 装配 effect 创建控制器 ctrl#1 并 `setTtsLifecycle(ctrl#1)`；
   `App.tsx:170-181` 开关 effect 见 `settings.handsFree=true` → `void ctrl#1.enable()`
   （async，内部要 `await vad.load()` + `getUserMedia` + `vad.start()`，
   `handsFreeController.ts:63-99`）。
3. StrictMode 触发 cleanup：`ctrl#1.disable()`（`App.tsx:163`）——但此刻 enable 还在
   await 中、`this.on` 仍为 false，`disable()` 第一行 `if (!this.on) return`
   （`handsFreeController.ts:102`）**直接返回，没有任何机制让在途的 enable 中止**。
4. remount 创建 ctrl#2，`setTtsLifecycle` 覆盖为 ctrl#2，开关 effect 再 `ctrl#2.enable()`。
5. ctrl#1 的 enable 随后完成：`this.on=true`、`vl.handsFreeOn()`（FSM 进 ARMED）、
   `startKws()`——**孤儿活控制器诞生**：自持 mic 流 + VAD + KWS，且其
   `deps.onSend=(t)=>sendRef.current(t)` 捕获的是同一个 App 实例的 ref，**发送通路完全有效**。

**首次唤醒的表现**（与用户观察逐点吻合）：
- 一声「小舟小舟」→ 两个 KWS 同时命中 → 两个 FSM 各自 wake → **两条唤醒提示音**
  （`playWakeCue` 是裸 `new Audio` 不排队，`audio.ts:308-318`——真·重叠播放）；
- 各开一条 `/api/asr/stream` → 同一句话出**两个 final** → `send()` 被调两次 →
  两个用户气泡、两轮后端请求；
- edge-gateway WS 循环是串行的（见 §2.2c），两轮回复先后到达，但 `pendingIdRef` 单槽
  被第二次 dispatch 覆盖 → 气泡归属错乱；TTS 层 `startTTSReply` 被连调两次后，两轮
  `speech_delta`/`final` 全部灌进同一个播放队列 → **同一答案播两遍**（与重叠的双提示音
  合起来即用户听到的「TTS 回复两个、重叠播报」）。
- **为什么只有首次双份**：`setTtsLifecycle`（`audio.ts:150-159`）是模块级单槽，只有
  ctrl#2 能收到 `ttsStart/ttsEnd`。ctrl#1 的 FSM 在第一轮 `onSend` 后进 THINKING，
  永远等不到 ttsEnd → **卡死 THINKING（THINKING 对 wake/VAD 全聋）**，此后不再参与
  ——之后每次唤醒只有 ctrl#2 响应。孤儿的 mic/VAD/KWS 仍在跑（白耗 CPU + 一路 AEC）。

**注意**：这不只是 StrictMode 的 dev 特产。「快速开→关→开 hands-free 开关」在生产
构建同样命中该竞态（disable 落在 enable 的 await 间隙即失效）。必须在控制器层修，
而不是关 StrictMode 了事。

### 2.2 U2 处理期间无法继续识别 —— 设计性全聋 + 四种死锁 + 网关无取消

**a) 设计性全聋（预期内，但要放宽）**：`voiceLoop.mjs` 中 THINKING 态：
`vadSpeechStart` 落 default 分支不动作（`voiceLoop.mjs:214-215`），`wake()` 无
THINKING 分支（`voiceLoop.mjs:187-197`）。普通指令 THINKING 只有 1-3s 可接受；但复杂
任务（深调研/多轮检索）THINKING 可达 90s，期间用户完全不可交互，这是体验硬伤。

**b) 四种 FSM 死锁（bug，比 a 严重）**：FSM 离开 THINKING 仅有 `ttsStart/ttsEnd` 两条腿
（`voiceLoop.mjs:279-287`），而以下四种情况 **App 从不调用 `ctrl.ttsEnd()`**
（`App.tsx` 的 final/error/watchdog 分支均无此调用；主卡 D4 写了「无音频时由 App 在
final 到达时调用」但从未实现）：
1. **TTS 关闭**（`ttsEnabled=false`）：final 到达后无人触发生命周期 → 卡死 THINKING，
   hands-free 一轮即废；
2. **回复无语音**（纯卡片，`data.speech` 空）：`finishTTSReply('')` 无分片入队 →
   `markTtsStart` 永不触发 → 卡死；
3. **error 事件**（`App.tsx:369-377`）：只画错误气泡，不通知控制器 → 卡死；
4. **看门狗超时**（`App.tsx:382-398`）：调了 `stopTTS()`，但 `stopTTS` 刻意不回放
   onEnd（`audio.ts:275-277` 注释「FSM 已自行迁移」——对 barge-in 成立，对看门狗不成立）
   → 卡死。
另有半种：TTS 全部合成失败（`finishTTSReply(...).catch` 静默，`App.tsx:348`）→ 从未
起播 → 卡死。

**c) 网关 WS 循环是串行的（实测代码，决定 P2 方案形态）**：
`gateway/edge/main.go:123-164`——`for { ReadMessage → orch.Handle → 内层 for 把事件流
drain 到 EOF }`。**一条请求的事件流没走完，不会读下一条 WS 消息**。含义：
- THINKING 期间就算 HMI 发新消息，也只是在 WS 缓冲区排队（最长等旧轮 90s ctx 走完），
  **不是打断**；
- 真「打断处理中」必须网关侧改为并发读 + 对在飞请求 `cancel()`——好消息是 ctx cancel 会
  沿 gRPC 天然传播到 edge-orchestrator → cloud → LLM 调用（通讯加固卡已做过预算级联），
  **零 proto 改动**；
- 顺带的正面事实：串行保证了事件不跨轮交错，HMI 侧用 FIFO 即可正确归属气泡（§2.6 A2）。

### 2.3 U3 唤醒态没有退出机制

现状 `dismissWords = ['没事', '不是叫你', '别听了', '没叫你']`（`voiceLoop.mjs:41`），
且 `includes(t)` **全等匹配**（`voiceLoop.mjs:265`）——「没事了」都不命中。用户说
「退下吧/再见/闭嘴/不用了」会被当正常请求发给 Planner，chitchat 还会礼貌地回一轮
（TTS 又开始说话）——与「退下」意图正相反，这是用户实测到的体验反差。

行业惯例（小爱/小艺/NOMI）：退下类指令**本地消化**——短应答（「好的」）→ 立即闭麦回
待机，唤醒词保持待命。注意与 D5-2 红线的边界：**确认条可见（needConfirm）时一切定稿
照发**（裸「取消/不要」必须上云走 F1 闭环）——exit 词只在无挂起确认时本地生效。

### 2.4 U4 唤醒后前几个字漏

三层拆解，**丢失全部发生在客户端**：

1. **KWS 检测延迟窗（约 0.2-0.5s，丢失不可逆）**：sherpa KWS 在唤醒词说完后经解码确认
   才回调；这段时间及其后到 ASR 就绪前的音频**没有任何缓存**——`vadEngine` 的
   AudioWorklet 帧只进推理不留存（`vadEngine.ts:78-92`），KWS 的 ScriptProcessor 同理。
   用户「小舟小舟帮我…」连着说时，「帮我」大概率落在这个窗里。
2. **wake→ASR 就绪 gap**：`_enterListening → openAsr → StreamingRecognizer.start`
   里 `rec.start(250)` 挂在 `ws.onopen` 之后（`audio.ts:398-404`）——WS 握手（本地几十
   ms，llm-gateway 冷时数百 ms）+ MediaRecorder 首片 250ms timeslice 之前**不采集**。
3. **服务端无损（已核实，不用改）**：llm-gateway 在 `start` 后建 ffmpeg + provider
   （`llm-gateway/http_server.py:226-241`），`pcm_queue` 无界缓冲、provider 未就绪期间
   音频不丢；qwen3 provider 的 session 建立期音频也在队列里等着。

**FOLLOWUP 续问漏字更狠**：免唤醒续问是 `vadSpeechStart` 才进 LISTENING/openAsr
（`voiceLoop.mjs:205-206`）——**开口之后**才开始建 ASR，首字丢失比唤醒路径更严重
（VAD speech-start 判定本身还有 64ms 去抖 + 帧延迟）。

修复分两级：P1 用「recorder 先行 + 本地分片缓冲」消掉第 2 层（纯前端，MediaRecorder
分片在 ws 未 open 时入队、onopen 发 start 后按序 flush——push-to-talk 同样受益）；
第 1 层（KWS 窗）只有**前滚缓冲（pre-roll）**能救，而 MediaRecorder 无法注入历史 PCM
→ P2 换 PCM 直传（§5.3）。

### 2.5 U5 语气词误收 + 半句截断

**5a 语气词**：定稿过滤仅 `graphemeLen(t) < 2`（`voiceLoop.mjs:264-265`）——单字
「嗯」能拦住，「嗯嗯/啊啊/哈哈/那个」全部上云，每个都换来 Planner 一轮正经回复。
持续聆听模式下这是高频噪声源。

**5b 半句截断，两个独立源**：
1. **qwen3 服务端硬编码 800ms**：`llm-gateway/providers.py:463`
   `turn_detection: {type: server_vad, silence_duration_ms: 800}` 写死；且 provider 收到
   首个 `.completed` 即 break 结束整条流（`providers.py:498-502`）。**含义：只要用 qwen3
   （默认引擎），客户端 `silenceTailMs` 设 1200 也无效**——停顿 800ms 必被定稿。
   fun-asr/mimo 不受此影响（它们的定稿由客户端 stop 驱动，`providers.py:570-580`、
   `providers.py:405-421`，客户端 VAD 是真权威——与 D3 设计一致的只有这两个引擎）。
2. **无续说合并**：人说话自然停顿（想词、报地名前犹豫）超过静音尾即被定稿发送，
   已发出去就没有任何挽回机制。

修复：P1 做**端点宽限合并**（engine 无关、纯前端）：定稿后不立即 send，若宽限窗内
VAD speech 再起 → 重开 ASR 把两段拼接（「导航去」+「西溪湿地」）；配**完整度启发式**
门控避免拖慢快指令（§5.2-3）。P2 把 `silenceTailMs` 透传给 qwen3（§5.3-2），治本 +
减少不必要的重开会话。宽限合并同时兜住 qwen3 提前定稿——asrFinal 到达时若 VAD 仍在
speech 中，直接判「未说完」，零额外等待。

### 2.6 审计发现（A1-A4）

- **A1 陈旧 ASR 回调劫杀下一轮**：`rec.onstop` 里挂的 7s 兜底 timer（`audio.ts:437`）
  在 `cleanup()` 后**不失效也不看代际**：LISTENING 被静默回收（falseWake 5s）→
  `closeAsr → cleanup()`（chunks 清空、finished 仍 false）→ 7s 到点 →
  `tryBatchFallback('识别超时')` → chunks 空 → `opts.onError` → 控制器
  `vl.asrFinal('')`（`handsFreeController.ts:177`）——**若此刻 FSM 已在下一轮
  LISTENING，空 final 会把新一轮打回 ARMED**（用户视角：刚唤醒又莫名退出）。
  另附 `ws.onmessage 'final'` 无 finished 守卫（`audio.ts:410`）：协议约定单 final，
  但 qwen3 异常路径有补发 final（`providers.py:507-512`），push-to-talk 的
  `onFinal` 直接 send（`Composer.tsx:81-85`）没有 FSM 状态挡板——理论双发。
- **A2 pendingIdRef 单槽**：两请求在飞时（U1 双发、或未来 P2 排队语义）第二次 dispatch
  覆盖 `pendingIdRef`（`App.tsx:423`），旧轮 final 会填错气泡、旧轮 speech 继续喂 TTS/
  确认条。网关串行（§2.2c）保证事件按轮有序 → **FIFO 队列即可完全正确归属**。
- **A3 引擎 start() 并发穿透**：`KwsEngine.start`/`VadEngine.start`/
  `StreamingRecognizer.start` 的 `if (this.running/active) return` 都在 async 函数首行，
  但 `running=true` 在**若干 await 之后**才置位（`kwsEngine.ts:99-126`、
  `vadEngine.ts:57-75`、`audio.ts:380-388`）——两个并发 start 都能穿过守卫，产生双
  ScriptProcessor/双 recorder（KWS 换词 `updateWakeKeywords` 的 stop+start 连击、
  设置快速开关时可命中）。
- **A4 D6 回声指纹部分是死代码**：`asrPartial` 的 SPEAKING 分支（`voiceLoop.mjs:237-240`）
  永不可达——进 THINKING 时已 `closeAsr`（`voiceLoop.mjs:136-140`），SPEAKING 期没有
  ASR 会话，何来 partial。真正在生效的回声防御=浏览器 AEC + barge-in 后 final 与 TTS
  文本重合校验（`voiceLoop.mjs:247-255`）。需修正主卡 D6 描述，避免后人以为有双保险。

## 3. 目标 / 非目标

**目标**
- G1：五个用户问题全闭环——首唤醒单份响应；处理中可打断（先解死锁、后真取消）；
  「退下吧」本地退场；唤醒即说不漏字（P1 显著缓解、P2 根治）；语气词不上云、
  自然停顿不截断。
- G2：语音回路在**四种终局**（正常播报/无语音回复/错误/超时）下都能回到可交互态，
  不存在永久卡死路径（FSM 兜底超时作最后防线）。
- G3：全部新逻辑延续 R4.3 纪律——**大脑在 `voiceLoop.mjs`（纯逻辑 node 可测）**，
  引擎/UI 只是外设；新增判定（exit 词/filler/宽限/完整度）全部有 node 测试。
- G4：默认行为零回归——hands-free 关闭时与今天逐字一致；push-to-talk 只受益
  （recorder 先行、final 守卫），不改交互。

**非目标**
- L4 真全双工（主卡 §2 已裁定非目标，理由不变）；

## 7. 后端改动边界（B 类清单——本文档的评审核心）

R4.3 主卡红线是「后端零改动」；本卡实测证明三个问题的根治点就在后端，故把口子**收敛为
以下 3 处加法**，评审通过即视为获批，超出即停：

| # | 服务 | 改动 | 性质 | 不做的代价 |
|---|---|---|---|---|
| B1 | llm-gateway `http_server.py` | ASR start 接受 `format:pcm16le` 直入 pcm_queue（跳 ffmpeg） | 加法，webm 路径不动 | U4 只能缓解不能根治（KWS 检测窗 0.2-0.5s 的字永远丢） |
| B2 | llm-gateway `http_server.py`+`providers.py` | start 接受 `vad_silence_ms` 透传 qwen3 session.update（替换硬编码 800） | 加法，缺省值=现状 800 | 默认引擎 qwen3 下客户端静音尾设置永远无效，长句必 800ms 截断（宽限合并可兜但每次多一轮 ASR 重开） |
| B3 | edge-gateway `main.go` | WS 并发读 + `{type:"cancel"}` 取消在飞请求 + `cancelled` 事件 | 加法，无 cancel 消息时行为不变；零 proto 改动 | THINKING 永远只能排队不能打断（复杂任务 90s 全聋） |

不碰：proto、编排核心（orchestrator/*）、registry、agents、memory。

## 8. 风险与回退

| 风险 | 缓解 |
|---|---|
| 宽限合并把两句独立话误拼（用户第一句说完立刻改口第二句） | 宽限窗短（700ms）+ 完整度启发式只对「悬挂结尾」开窗；拼接前缀在下轮定稿 partial 上屏可见，用户可感知；参数可调可禁（endpointGraceMs=0 即关） |
| exit 词误杀正常请求（「结束导航」≠退下） | 前缀匹配词表刻意不含单独「结束」；「结束导航/停止导航」不以 exitWords 开头，不命中；node 测覆盖边界词 |
| PCM 直传后批处理兜底路径回归 | 兜底改 WAV 包装同样走既有 `/api/asr`，e2e_voice_loop 扩 PCM round-trip 护栏 |
| B3 并发化引入网关竞态（双请求写同一 WS） | `client.send` 已有互斥；每连接同时至多一个在飞请求（新请求先 cancel 旧的），语义反而比今天更明确 |
| 取消传播到半执行的动作 | 车控在端侧快路径毫秒级完成不进 THINKING；云侧长尾均为只读检索/生成；交易域量产前需「不可取消段」（记待办，PoC 均 mock） |
| StrictMode 修复只治 dev 不治本 | 修在控制器 epoch 层，生产构建下「快速开关开关」同样被护住；不关 StrictMode |
| qwen3 `vad_silence_ms` 上限外行为未知 | 夹紧 [300,2000]；改动带 env 默认值兜底，异常回落 800 现状 |
