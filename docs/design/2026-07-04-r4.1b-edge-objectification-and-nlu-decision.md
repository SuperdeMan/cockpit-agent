## 1. 为什么有这张卡（R4.1 P3 收官暴露的事实）

R4.1 P3 做了两批**纯 pattern、零 VAL 对象改动**的 quick-win，覆盖率 72.04%→75.74%（+3.70pt）：

| 批 | 内容 | 增量 |
|---|---|---|
| B1 | 气象并入天气类（同义词） | +2.02pt |
| B2 | 设置页/界面开合族 + 修既有劫持 | +1.68pt |

**但 82% 目标按母卡 B1/B2/B3 计划到不了**，差 ~6pt（~530 句）。根因是母卡 §6 对 B2 大头
「辅助驾驶 482」的前提读偏：实测辅助驾驶漏判 309 条**不是「打开 XX 设置页面」pattern**，
而是 **ADAS 功能开关长尾**——每个功能一个词、各 ≤12 句、共 12+ 种（见 §2）。这类要端侧
可执行，**每个都得补一个独立 VAL 对象**（`commands.yaml` 对象 + `entities`/`responses` 话术
+ `LOCAL_INTENTS` + `fast_intent` 规则 + 保证 VAL 结构化执行链跑通），**不是「扩规则」，是
「补对象」**。B3 导航播报（`navigation` 是 `online_only`，端侧播报开关需新建 `navi_broadcast`
对象）、空气净化同理。

**这正是母卡 §7 K6 的触发信号**：「扩规则吃完易摘果实后边际递减、剩余是对象密集型长尾」。

## 2. 待对象化的缺口清单（量化）

数据来自 P2 全量语料覆盖率报告（`--corpus full`），当前总体 **75.74%**、端侧应接子集
（`edge_expected!=false`）**~77.6%**。剩余端侧应接的主要漏判对象：

| 对象 | 漏判量级 | 性质 | 对象化成本 |
|---|---|---|---|
| 辅助驾驶（ADAS 长尾） | ~309（12+ 种功能：限速提示/手持电话监测/危险驾驶监测/抽烟监测/行人低速提示音/车距检测预警/乘客区安全带提醒/车道纠偏/前方交通穿行预警/预测型碰撞报警…） | 功能开关（on/off），每种一个对象 | **高**：12+ 个 VAL 对象；但**同构**（都是 on/off toggle），可模板化批量生成 |
| 界面/页面残余 | ~部分（B2 已接住带「设置/界面/页面」的；剩下的是无这些词的专有 UI 页） | 页面开合 | 低-中：多数已被 B2 通用兜底接住 |
| 声音/语音助手残余 | ~按键音/媒体音量/音区/发音人切换 | 控制/设置 | 中：部分需新对象（音区/发音人） |
| 导航播报（B3） | ~24（edge_expected=true 子集） | 端侧 TTS 播报开关 | 低：1 个 `navi_broadcast` 对象 |
| 空气净化 | 2 | on/off | 低：1 个 `air_purifier` 对象 |

**关键判断**：辅助驾驶 ADAS 虽量大，但**高度同构**（清一色 on/off toggle）。若走 A 路径，
不该逐个手写 12+ 个对象，而应**数据驱动批量生成**（一张 ADAS 功能表 → 脚本生成 commands.yaml
对象 + fast_intent 规则），复用 P2 已建的 gen 脚本范式。

## 3. 两条路径

### A · 端侧对象化（继续扩规则思路，数据驱动批量补对象）

- 做法：把 ADAS 功能长尾 + 导航播报 + 空气净化 + 声音/语音残余**数据驱动**补成 VAL 对象
  （一张功能表驱动脚本生成 commands.yaml/规则/话术，人工审核同构模板），配黄金用例 + 反例
  （每对象「怎么用=manual 云端」「XX监测的原理=云端」不误接）。
- 收益：覆盖率可望冲 82%+（ADAS ~309 是最大单块）。
- 成本：M-L。真正的活不在「写规则」而在**保证每个新对象在 VAL 结构化执行链（归一化→校验→
  安全门控→模拟→话术）真能落地**，且不与既有对象冲突；ADAS 功能的**安全门控**（行车中能否
  开关某辅助功能）需逐个甄别（碰 `ws8` 安全门控，属架构敏感区）。
- 风险：碰 VAL 执行链 + 安全门控；对象数从 62→80+ 后 fast_intent 规则表膨胀（母卡 §7 的第二个
  NLU 触发条件「模式表突破 ~300 条且单批回归互扰」逼近）。

### B · 端侧轻量 NLU 模型（母卡 §7 路径 2 / 架构 §3.3）

- 做法：训练/接入端侧轻量意图分类器替换/兜底规则引擎。**语料已由 P2 资产化**
  （`feishu_intents_full.jsonl` 8590 条真实说法 + 词库实体）= 现成训练/评测集；P2 覆盖率报告
  = 现成评测基线（NLU vs 规则同一把尺子直接比）。
- 收益：一次性解决长尾（不用逐对象补规则）；架构上更干净（D7 规则/引擎不分离的债一并偿）。
- 成本：L-XL。需评估端侧模型体积/延迟/离线可用性（架构 §3.3 端侧「快系统」约束）、训练管线、
  与 VAL 执行链的衔接（NLU 出意图后仍要落 commands.yaml 对象执行——**NLU 解决「识别」不解决
  「执行」，执行侧对象化的活 A 路径那部分照样要做**）。
- 风险：端侧模型工程化是新领域（架构 §3.3 列为可选项 + 风险 R1「先做端侧模型基准测试」）。

### 关键洞察：A 与 B 不是二选一的对立

**NLU 只解决「识别」（意图分类），不解决「执行」（VAL 对象化）**。哪怕上了 NLU，识别出
「打开车道偏离辅助」后，端侧仍需一个可执行的 ADAS 对象。所以：
- **执行侧对象化**（补 VAL 对象让端侧能真正执行）是**两条路径都要做**的地基。
- **识别侧**才是 A（继续扩 fast_intent 规则）vs B（NLU 模型）的真正分岔。

## 4. 推荐

**先做「执行侧对象化」的地基（两路共用），识别侧 A/B 暂缓、用数据说话**：

1. **P0（本卡）**：先补**低成本、非安全敏感**的三个对象——`navi_broadcast`（导航播报，
   edge=true 子集）+ `air_purifier`（空气净化）+ 声音/语音的 1-2 个明确对象。数据驱动、配反例，
   走完整 B1/B2 四重护栏。预期 +1~2pt（把「稳赢的小对象」清掉）。
2. **P1（本卡）**：ADAS 功能表**数据驱动批量对象化**，但**安全门控逐个甄别**（行车中禁开关
   哪些辅助）——这是碰 ws8 的活，须 Plan Mode 对齐架构后再动。预期 +3~4pt（冲 82%）。
3. **识别侧 A/B 决策**：P1 做完再看——若规则表已逼近 ~300 条且单批回归开始互扰（母卡 §7 第二
   触发条件），或覆盖率仍卡在 <85%，则启动 B（NLU）重评估，语料/基线现成。否则 A 继续够用。

**理由**：执行侧对象化是绕不开的地基（NLU 也要它）；先清低成本对象拿确定收益；ADAS 因碰安全
门控值得单独 Plan Mode；识别侧 A/B 是**数据触发的决策**，不预先押注（母卡 §7 的纪律）。

## 5. 非目标

- 本卡不实现任何对象（待选路径 + Plan Mode 后另起实现卡）。
- 不改语义路由（R4.1 P0 + 重排已收口）。
- 不追求 100% 覆盖（导航搜索/规划类本就上云，母卡 §5.3 已甄别）。

## 6. 验收（当选定路径、进入实现时）

- 每新增对象：黄金用例 + 反例（怎么用/原理类不误接）入 `edge_regressions.yaml`；覆盖率报告
  提升且 ≥ 该批预期；`--corpus full` / 39→N 策展 / smoke 13 / 全量 pytest 四重护栏零回归；
  **VAL 结构化执行真栈验证**（不是只让 fast_intent 识别，要 edge_call→VAL→action 真跑通）。
- ADAS 批：安全门控（行车禁开关）逐功能甄别记录，`ws8` 回归全绿。
